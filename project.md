# Bot Funnel Service — План робіт MVP

## Що це за продукт

Сервіс для побудови мультиканальних воронок у месенджерах. **На MVP — тільки Telegram.** Користувачі реєструються, створюють проекти, підключають своїх Telegram-ботів і будують воронки спілкування з підписниками. Підтримуються масові розсилки та публічне API для інтеграцій з зовнішніми системами.

**Кінцева мета v1.0 (MVP):** користувач за 10 хвилин підключає свого бота, будує лінійну воронку з 3-5 кроків, запускає першу розсилку на 100+ підписників.

## Ролі та доступи

### 1. Super Admin (адміністратор платформи)
Технічний / власник сервісу. 1-2 особи на всю систему. **Не реєструється через звичайну форму** — створюється seed-скриптом або вручну в БД.

**Може:**
- Бачити список усіх користувачів сервісу
- Бачити список усіх проектів незалежно від власника
- Переглядати глобальну аналітику (кількість користувачів, активних ботів, повідомлень за добу)
- Блокувати / розблоковувати користувача
- Видаляти проект цілком (з усім вмістом)
- Бачити журнал ключових подій (логін, створення проекту, помилки)

**Не може:**
- Редагувати чужі воронки чи розсилки (read-only по контенту)
- Логінитись у чужий обліковий запис (без імперсонації на MVP)
- Бачити токени Telegram-ботів у відкритому вигляді (тільки masked)

### 2. User (звичайний користувач)
Реєструється самостійно через email + пароль. Кожен користувач — власник одного або декількох власних проектів.

**Може:**
- Створювати / редагувати / видаляти власні проекти
- У межах свого проекту: керувати ботом, підписниками, воронками, розсилками, API-ключем

**Не може:**
- Бачити чужих користувачів і чужі проекти

### 3. Project Owner = User-creator
На MVP кожен проект має одного власника — це той, хто його створив. Без можливості додавати співвласників (це винесено в `12-nice-to-have`).

### 4. Subscriber (підписник)
Кінцевий користувач Telegram, який спілкується з ботом. **Це не акаунт у нашому сервісі.** Зберігається у межах конкретного проекту. Один і той самий Telegram-юзер у двох різних проектах = два різних Subscriber-записи (свідома ізоляція).

## Архітектура сутностей

```
User
 └── Project (1..N, до 5 на старті)
      ├── Bot (1, на MVP практично завжди один)
      ├── Subscriber (N) — заповнюється автоматично з Telegram
      ├── Funnel (N)
      ├── Broadcast (N)
      ├── Custom Field (N — схема)
      └── API Key (1 primary)
```

## Послідовність виконання

Рекомендований порядок (щоб задачі не блокували одна одну):

1. **`01-фундамент`** — старт, налаштовує все інше
2. **`02-автентифікація`** + **`03-проекти`** — паралельно після фундаменту
3. **`04-телеграм-боти`** — після проектів
4. **`05-підписники`** — після ботів (підписники з'являються через бот)
5. **`06-воронки`** + **`07-розсилки`** — паралельно після підписників
6. **`08-публічне-api`** — коли вже є підписники / воронки / розсилки (API їх експонує)
7. **`09-аналітика`** + **`10-адмін-панель`** — наприкінці, бо потребують готових даних
8. **`11-фронтенд`** — паралельно з backend для кожного епіка (ідеально один frontend-developer бере епіки в роботу за 2-3 дні після backend)

**Орієнтовний обсяг для одного fullstack-розробника:** 8-10 тижнів до робочого MVP.

## Принципи MVP (чого свідомо НЕ робимо)

- Один монорепо, один backend-сервіс, одна БД (PostgreSQL). Без мікросервісів.
- Без Kubernetes, Helm, Service Mesh — Docker Compose на одному VPS.
- Воронки **лінійні** з простими розгалуженнями за натиснутою кнопкою. Без drag-and-drop canvas.
- Один канал — Telegram. Viber, Instagram, WhatsApp — у `12-nice-to-have`.
- Аналітика — три цифри на воронку, три на розсилку. Без BI, cohort'ів, custom dashboards.
- UI — функціональний, не дизайнерський. Tailwind + shadcn/ui-готові компоненти.
- Без платних тарифів і білінгу — фокус на product-market fit.
