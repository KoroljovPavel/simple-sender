## Description

Full-feature test quality audit for the foundation infrastructure feature. The audit verifies that all three integration tests (`HealthEndpointTest`, `HealthSecurityTest`, `SecurityBlockTest`) have meaningful assertions, are correctly set up with minimal `@MockBean` usage, and would actually catch regressions — i.e., they fail if the production code is broken.

The audit is the final quality gate on tests before the Pre-deploy QA wave. Unlike reviewer roles (code-reviewer, test-reviewer) that review individual tasks, this audit takes a holistic view of the entire test suite produced by the feature.

Key concern: tests that always pass regardless of what the code does provide false confidence. This audit explicitly checks whether each test would fail under specific regression scenarios.

## Details

**Test files to audit:**

- `backend/src/test/java/com/botfunnel/HealthEndpointTest.java` — primary endpoint test: HTTP 200 + body assertion
- `backend/src/test/java/com/botfunnel/HealthSecurityTest.java` — security permit test: `/health` accessible without auth
- `backend/src/test/java/com/botfunnel/SecurityBlockTest.java` — security block test: undefined paths return 401

**@MockBean correctness details:**

The correct class to mock is `com.mongodb.reactivestreams.client.MongoClient`. Spring Data MongoDB reactive auto-configuration requires this interface to be present to initialize. A common mistake is trying to mock `ReactiveMongoClient` — that class does not exist; only `MongoClient` from `com.mongodb.reactivestreams.client` is the real reactive client interface. If the wrong class is mocked, the Spring context will fail to start (or may not suppress the auto-config).

`ReactiveRedisConnectionFactory` (from `org.springframework.data.redis.connection`) must also be mocked to prevent Lettuce from attempting a real Redis connection during context startup.

**Regression sensitivity checks — what to look for:**

For each test, mentally simulate the failure scenario:
- If the test only calls `.exchange()` without `.expectStatus().isOk()` → it won't catch HTTP 500
- If the test checks `.expectBody(String.class).isEqualTo("{\"status\":\"ok\"}")` (exact string) instead of JSON path → brittle (whitespace-sensitive), but not wrong; note as minor style finding
- If `HealthSecurityTest` adds `.header("Authorization", "Bearer xxx")` → it's not testing unauthenticated access

**patterns.md deviation (Decision 6):**

`patterns.md` states: "No mocking of the database in integration tests — use real embedded MongoDB." This feature intentionally deviates because the Epic 01 tests exercise only HTTP routing and security filters, never touching MongoDB queries. The deviation is documented in Decision 6 of the tech-spec. The auditor must note this deviation as accepted — not flag it as a violation. All future epics with real repository usage must follow the original patterns.md rule.
