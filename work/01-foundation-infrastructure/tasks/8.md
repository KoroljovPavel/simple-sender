# Task 8: Code Audit

## Required Skills

Before starting, load: `/skill:code-reviewing` — [SKILL.md](../../.claude/skills/code-reviewing/SKILL.md)

## Description

Full-feature code quality audit for the 01-Foundation Infrastructure feature. The auditor reads every source file created across all seven implementation tasks and reviews the codebase holistically — not file-by-file in isolation, but as a coherent whole.

This task is purely analytical. The auditor does NOT fix issues. If issues are found, the execution lead reads the audit report and spawns a dedicated fixer agent for each issue. The auditor writes a structured report to `work/01-foundation-infrastructure/logs/working/task-8/code-audit-report.md` and a summary to `decisions.md`.

The audit covers the complete surface area of the feature: Spring Boot backend (Gradle build, application wiring, SecurityConfig, HealthController, integration tests), Nuxt 4 frontend skeleton, Docker Compose infrastructure, shell scripts, and developer documentation.

## What to do

1. Read all source files listed in the Context Files section below (all files created in this feature).

2. Audit Spring Boot conventions:
   - `BotFunnelApplication.java` — correct `@SpringBootApplication` placement, package structure matches `com.botfunnel` root
   - `HealthController.java` — `@RestController` annotation, returns `Mono<Map<String, String>>`, correct mapping (`@GetMapping("/health")`)
   - `common/package-info.java` — placeholder is present and well-formed
   - `application.properties` — no hardcoded secrets, reasonable defaults, no redundant properties

3. Audit SecurityConfig correctness (highest-priority area per tech-spec Decision 5):
   - Both `@Configuration` and `@EnableWebFluxSecurity` are present (note: `@EnableWebFluxSecurity` does NOT include `@Configuration` in Spring Security 6)
   - `SecurityWebFilterChain` bean is defined via `ServerHttpSecurity`
   - CSRF is explicitly disabled (stateless REST API)
   - Rule ordering: `pathMatchers("/health").permitAll()` appears BEFORE `anyExchange().authenticated()` — swapping these breaks the feature
   - No unintended open paths (only `/health` is permitted without auth)

4. Audit Gradle build hygiene:
   - `settings.gradle` — project name set correctly, Foojay resolver plugin present for JDK auto-provisioning
   - `build.gradle` — Spring Boot 3.5.x, Java 21 toolchain, all required starters present (`webflux`, `security`, `data-mongodb-reactive`, `data-redis-reactive`), `devtools` is `developmentOnly`, test dependencies in `testImplementation` scope
   - No version pinning inconsistencies, no unnecessary dependencies, no deprecated configurations

5. Audit integration tests:
   - `HealthEndpointTest` — asserts HTTP 200 and body `{"status":"ok"}`, uses `@SpringBootTest(RANDOM_PORT)` + `WebTestClient`
   - `HealthSecurityTest` — same request without auth → HTTP 200 (verifies SecurityConfig permits `/health`)
   - `SecurityBlockTest` — `GET /api/nonexistent` without auth → HTTP 401 (verifies `anyExchange().authenticated()`)
   - `@MockBean` usage: mocks `MongoClient (com.mongodb.reactivestreams.client.MongoClient)` and `ReactiveRedisConnectionFactory` — minimal set, no over-mocking
   - Tests would actually FAIL if the security config were broken or the endpoint returned wrong data

6. Audit Docker Compose practices:
   - `mongo:8.0` and `redis:7.4` official images used
   - Ports bound to `127.0.0.1` (not `0.0.0.0`) — prevents network exposure on local dev
   - Named volumes `mongodb_data` and `redis_data` for data persistence across restarts
   - No application containers in this file (backend/frontend run locally)
   - Valid compose file structure, no deprecated fields

7. Audit shell script quality (`scripts/install-hooks.sh`):
   - Script is idempotent (safe to run multiple times)
   - Script is executable (`chmod +x` or correct permissions)
   - Copies hook to `.git/hooks/pre-commit` and makes it executable
   - Soft failure when gitleaks is not installed (exits 0 with warning, does not block commits)
   - No unsafe bash constructs (`set -e` or equivalent error handling if appropriate)

8. Audit documentation completeness (`docs/local-setup.md`):
   - Prerequisites section covers: Java 21, Node 24, Docker, pnpm, gitleaks
   - Step-by-step setup matches actual project commands (install hook, copy .env, docker compose, bootRun, pnpm dev)
   - Port conflict resolution documented for all four ports: 27017, 6379, 8080, 3000
   - Windows/WSL2 warning present
   - Verification checklist present (at minimum: `GET /health` check)

9. Audit root files:
   - `.gitignore` — covers all required patterns: `backend/build/`, `backend/.gradle/`, `frontend/node_modules/`, `frontend/.nuxt/`, `.env`, `*.key`, `secrets/`, `.idea/`, `*.pem`, `*.p12`, `*.jks`
   - `.env.example` — exactly two variables (`MONGODB_URI`, `REDIS_URL`) with inline comments explaining each; no real values
   - `.nvmrc` — contains Node 24 version string

10. Audit frontend skeleton:
    - `nuxt.config.ts` — `compatibilityDate` field present, TypeScript strict mode configured, `ssr` is NOT set to `false`
    - `app.vue` — minimal Hello World, valid Vue 3 SFC
    - `package.json` — nuxt 4.4.x dependency, pnpm as package manager

11. Write the audit report to `work/01-foundation-infrastructure/logs/working/task-8/code-audit-report.md` with:
    - Executive summary (overall quality, key findings count)
    - Findings by area: severity (CRITICAL / WARNING / INFO), description, affected file, recommended fix
    - Verdict: PASS (no blockers) or FAIL (blockers present)

## Details

**Audit focus areas by priority:**

1. **CRITICAL — SecurityConfig correctness:** This is the highest-risk area in Epic 01. The decision to configure security upfront (Decision 5) is sound but the implementation must be exactly right. Wrong rule ordering (anyExchange before pathMatchers) causes 401 on `/health`. Missing `@Configuration` alongside `@EnableWebFluxSecurity` causes Spring not to pick up the config. Missing CSRF disable causes 403 on non-GET requests in future epics.

2. **HIGH — @MockBean strategy:** The deviation from patterns.md (Decision 6) is intentional and approved. Verify the mock is the narrowest possible scope: only `MongoClient` and `ReactiveRedisConnectionFactory` are mocked, nothing else. Over-mocking would hide future integration issues.

3. **MEDIUM — Gradle hygiene:** Verify no transitive dependency conflicts, `developmentOnly` scope for devtools is correct (Spring Boot Gradle plugin adds this configuration — verify it's declared before use), toolchain declaration format is current Gradle 8.x style.

4. **MEDIUM — Docker security defaults:** `127.0.0.1` binding is a deliberate security choice (Decision 9). Validate both MongoDB and Redis ports use this binding, not the default `0.0.0.0`.

5. **LOW — Documentation accuracy:** `docs/local-setup.md` accuracy check — does the documented command sequence match the actual files? Specifically: is the `docker compose` command using the correct `-f infra/docker-compose.yml` path?

**Edge cases to check:**
- `HealthController` package: must be `com.botfunnel` (root package), not a sub-package — otherwise Spring component scan may miss it depending on `@SpringBootApplication` placement
- `application.properties` — if `server.port` is not set, Netty defaults to 8080 (correct). If it IS set, verify it's 8080
- Nuxt `compatibilityDate` format — must be a valid date string (e.g., `"2024-04-03"`), not just any string
- `install-hooks.sh` idempotency — running twice must not corrupt `.git/hooks/pre-commit`

**Known approved deviations from patterns.md (do NOT flag as issues):**
- `@MockBean` for DB in integration tests — approved per Decision 6
- Spring Boot 3.5.x (not 4.x) — approved per Decision 1
- Nuxt 4 (not Nuxt 3) — approved per Decision 2
