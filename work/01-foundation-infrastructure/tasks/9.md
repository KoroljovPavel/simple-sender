# Task 9: Security Audit

## Required Skills

Before starting, load: `/skill:security-auditor` — [SKILL.md](~/.claude/skills/security-auditor/SKILL.md)

## Description

Full-feature security audit of the Epic 01 foundation infrastructure. This task runs after all implementation waves are complete and examines every security-relevant artifact created in this feature.

The core concern for Epic 01 is misconfiguration: Spring Security WebFlux without an explicit `SecurityWebFilterChain` blocks everything; with a misconfigured one it can silently open all paths. Equally, a leaked credential in `.env.example`, an overly permissive Docker Compose port binding, or a bypassable gitleaks hook all represent security failures that are cheapest to catch before the project grows.

The audit applies OWASP Top 10 principles in the context of a local developer infrastructure setup: A01 (Broken Access Control — Spring Security rules ordering), A02 (Cryptographic Failures — no secrets in tracked files), A05 (Security Misconfiguration — Docker Compose defaults, CSRF policy), and A07 (Identification and Authentication Failures — security filter correctness).

The auditor reads each artifact, checks it against the criteria below, and writes a structured report to `work/01-foundation-infrastructure/logs/working/task-9/security-audit-report.md`.

## What to do

- [ ] **Audit SecurityWebFilterChain** — read `backend/src/main/java/com/botfunnel/security/SecurityConfig.java`: verify `@Configuration` and `@EnableWebFluxSecurity` are both present; CSRF is explicitly disabled; `pathMatchers("/health").permitAll()` appears BEFORE `anyExchange().authenticated()` (first-match ordering); no additional `permitAll()` rules that open unintended paths
- [ ] **Audit `.gitignore`** — verify coverage of: `.env`, `*.key`, `secrets/`, `.idea/`, `*.pem`, `*.p12`, `*.jks`, `backend/build/`, `backend/.gradle/`, `frontend/node_modules/`, `frontend/.nuxt/`; confirm no build artifacts are tracked in git
- [ ] **Audit `.env.example`** — verify it contains ONLY placeholder/example values (no real credentials); check that only `MONGODB_URI` and `REDIS_URL` are present (minimal scope for Epic 01); verify inline comments are present for each variable
- [ ] **Audit `scripts/install-hooks.sh`** — check for command injection vulnerabilities in variable usage; verify the soft-failure pattern (exits 0 with warning when gitleaks not installed) is correct and not a security bypass; confirm script is idempotent; check that the installed hook itself cannot be trivially bypassed
- [ ] **Audit `infra/docker-compose.yml`** — verify both MongoDB and Redis ports are bound to `127.0.0.1` (not `0.0.0.0`); confirm no hardcoded credentials in the compose file; verify named volumes are used (data does not leak to uncontrolled host paths); check image versions are pinned (`mongo:8.0`, `redis:7.4`)
- [ ] **Audit `docs/local-setup.md`** — verify no real credentials appear in examples; confirm gitleaks installation instructions are present; check that the soft-failure behavior of the hook is documented (developer aware that not installing gitleaks means no protection)

## Details

**Files to audit and what to check:**

- `backend/src/main/java/com/botfunnel/security/SecurityConfig.java` — Spring Security 6 WebFlux. Key risk: annotation completeness (`@EnableWebFluxSecurity` does NOT include `@Configuration` in Spring Security 6 — both required explicitly). Rule ordering is first-match: if `anyExchange().authenticated()` appears before `pathMatchers("/health").permitAll()`, all requests including `/health` will require auth. CSRF must be explicitly disabled — Spring Security 6 enables it by default for WebFlux, which breaks all non-GET requests with 403.

- `.gitignore` — Check both the patterns exist AND that no sensitive files are currently tracked in git (`git ls-files | grep -E '\.(env|key|pem|p12|jks)$'` is a useful check). Missing `.idea/` is a common oversight that leaks developer IDE config.

- `.env.example` — Zero-tolerance for real credentials. Any value that looks like a real URI with credentials (e.g., `mongodb://user:pass@host`) is a finding. Expected values: `mongodb://localhost:27017/botfunnel` and `redis://localhost:6379` (or equivalent localhost placeholders).

- `scripts/install-hooks.sh` — Variable expansion risks: unquoted `$VARIABLE` in paths could allow injection. The soft-failure pattern (gitleaks not found → exit 0 with warning) is an **accepted risk** documented in tech-spec Risk table — do NOT flag as vulnerability, but verify it is documented in `docs/local-setup.md`. Check idempotency: running the script twice should not corrupt the hook.

- `infra/docker-compose.yml` — Default Docker port binding is `0.0.0.0` (all interfaces) which exposes unauthenticated MongoDB and Redis to the local network. The tech-spec explicitly requires `127.0.0.1:27017:27017` syntax. This is a concrete finding if missing. No `environment:` block with `MONGO_INITDB_ROOT_*` or `REDIS_PASSWORD` expected (local dev convenience — document as accepted risk if ports are correctly bound to localhost).

- `docs/local-setup.md` — Check all code examples and commands for accidentally included real values. Verify the gitleaks section explains what happens if it is not installed (soft failure = no protection until installed).

**Accepted risks (from tech-spec — do NOT re-flag):**
- Gitleaks soft-failure exits 0: intentional, documented. Risk accepted because no CI-side compensating scan in Epic 01. Mitigation: install instructions in docs.
- No auth on MongoDB/Redis: mitigated by localhost-only port binding. No password configured for local dev convenience.

**Report format:** Write structured Markdown with sections: Summary (pass/fail per area), Findings (severity: CRITICAL / HIGH / MEDIUM / LOW / INFO), Accepted Risks, Recommendations.
