# Task 5: Registration + email verification + resend

## Required Skills

Before starting, load:
- `/skill:code-writing` — [SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Implement registration (`POST /api/auth/register`), email verification (`GET /api/auth/verify-email`), and resend verification (`POST /api/auth/resend-verification`). Finalize `TokenService` with `SecureRandom`, base64url encoding, and SHA-256 hashing. Add the `@ValidPassword` custom Bean Validation constraint used by registration and change-password DTOs.

Registration flow: validate → check email uniqueness (including 30-day soft-delete window) → BCrypt hash password → create pending User → generate raw token → store SHA-256(token) in user document → send verification email fire-and-forget → return 201.

Token security (Decision 2): raw token is sent only in email, SHA-256 hash stored in DB. Even if MongoDB is compromised, raw tokens cannot be extracted.

## What to do

1. **Finalize `TokenService`** at `backend/src/main/java/com/botfunnel/auth/TokenService.java`:
   - `generateRawToken()`: `SecureRandom.nextBytes(32)`, `Base64.getUrlEncoder().withoutPadding().encodeToString(bytes)` → returns 43-char URL-safe string
   - `hashToken(String rawToken)`: `MessageDigest.getInstance("SHA-256").digest(rawToken.getBytes(StandardCharsets.UTF_8))` → hex-encoded string

2. **Create `@ValidPassword`** at `backend/src/main/java/com/botfunnel/auth/validation/ValidPassword.java` (annotation + `ValidPasswordValidator.java`):
   - Rule: min 8 chars, at least 1 letter (`[a-zA-Z]`), at least 1 digit (`[0-9]`)
   - `isValid()` regex: `password.length() >= 8 && password.matches(".*[a-zA-Z].*") && password.matches(".*[0-9].*")`

3. **Create registration DTO** `RegisterRequest` (`@NotBlank @Email String email`, `@NotBlank String name`, `@NotBlank @ValidPassword String password`) in `backend/src/main/java/com/botfunnel/auth/dto/`.

4. **Extend `AuthService`** with registration logic:
   - `register(RegisterRequest request, ServerWebExchange exchange)` → reactive chain:
     1. Normalize email to lowercase
     2. Find user by email
     3. If found and `status != deleted` → throw `AppException.conflict("Користувач з таким email вже існує")`
     4. If found and `status == deleted` and `deletedAt > now - 30 days` → throw `AppException.conflict("Акаунт... зверніться до підтримки: {SUPPORT_EMAIL}")`
     5. If not found (or deleted > 30 days ago): BCrypt-hash password, create User with `status=pending`, `createdAt=now`, `updatedAt=now`
     6. Generate rawToken via `tokenService.generateRawToken()`, store `tokenService.hashToken(rawToken)` in `user.emailVerificationTokenHash`, set `emailVerificationExpiresAt = now + 24h`
     7. Save user
     8. Send verification email fire-and-forget via `emailService.sendVerificationEmail(email, name, rawToken)` — errors logged, never propagated
     9. Return 201 with user ID
   - `verifyEmail(String rawToken, ServerWebExchange exchange)`:
     1. Hash the rawToken
     2. Find user by hash where `status IN (pending, active)` only — **no expiry filter in the DB query**
     3. If not found: throw `AppException.badRequest("Посилання недійсне")`
     4. If found but `expiresAt <= now` (check expiry in the service layer after retrieving the user): throw `AppException` with code `TOKEN_EXPIRED`
     5. Update: `status=active`, clear token fields, `updatedAt=now`
     6. Log `email_verified` event
     7. Return `{redirect: "/login"}`
   - `resendVerification(String email)`:
     1. SET NX EX 60 Redis key `resend:rate:{email}` — always, regardless of whether user exists (prevents timing oracle)
     2. `setIfAbsent()` returns `Mono<Boolean>` — `Boolean.FALSE` means the key already existed (rate limited); check with `.map(success -> !success)` to detect this case; throw `AppException.tooManyRequests(...)` if rate limited
     3. Find user by email; if exists and `status=pending`: generate new token, update hash and expiry, save, send email fire-and-forget
     4. Always return 200 (never reveal if email exists)

5. **Extend `AuthController`** with:
   - `POST /api/auth/register` → `@Valid @RequestBody RegisterRequest` → call `authService.register()` → 201
   - `GET /api/auth/verify-email?token=` → call `authService.verifyEmail()` → 200 with redirect field
   - `POST /api/auth/resend-verification` → call `authService.resendVerification()` → 200

**Edge cases:**
- Soft-delete check: `status == deleted AND deletedAt > now - 30 days`. User deleted 31+ days ago is NOT in the 30-day window and can register with the same email again.
- `ReactiveRedisTemplate.opsForValue().setIfAbsent(key, "1", Duration.ofSeconds(60))` returns `Mono<Boolean>` — false if key existed (rate limited), true if key was set (proceed)
- Token lookup: `UserRepository.findByEmailVerificationTokenHash(hash)` must check both status and expiry; either filter in query or in service layer

**Implementation hints:**
- `MessageDigest.getInstance("SHA-256")` is not thread-safe; create a new instance per call: `MessageDigest.getInstance("SHA-256").digest(token.getBytes(StandardCharsets.UTF_8))` and hex-encode the result. Do NOT use `org.springframework.util.DigestUtils` — it only provides MD5, not SHA-256. If you prefer a utility method, `org.apache.commons.codec.digest.DigestUtils.sha256Hex()` is available transitively via spring-boot-starter-test
- Base64url with no padding: `Base64.getUrlEncoder().withoutPadding()` — standard Java API, no external library
- For Mailpit token extraction in tests: use Mailpit API (`/api/v1/messages` → get latest message, extract body, find token URL with regex)
