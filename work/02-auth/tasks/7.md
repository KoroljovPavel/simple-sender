# Task 7: Profile endpoints + Super Admin seed + JobRunr hard-delete

## Required Skills

Before starting, load:
- `/skill:code-writing` — [SKILL.md](~/.claude/skills/code-writing/SKILL.md)

## Description

Implements three thematically related components: profile CRUD endpoints, an idempotent Super Admin seeder, and a recurring hard-delete job. The `change-password` endpoint invalidates all sessions except the current one — for the terminate-all-sessions query, coordinate with Task 6 to use the verified sessions collection field path.

## What to do

1. **Create `ProfileController`** at `backend/src/main/java/com/botfunnel/profile/ProfileController.java`:
   - `GET /api/profile` — return current user's name, email, status
   - `PATCH /api/profile` — accept `UpdateProfileRequest` (name only); return updated profile
   - `POST /api/profile/change-password` — accept `ChangePasswordRequest`; return 200
   - `POST /api/profile/terminate-all-sessions` — return 200
   - `DELETE /api/profile` — return 200

2. **Create `ProfileService`** at `backend/src/main/java/com/botfunnel/profile/ProfileService.java`:
   - `getProfile(String userId)` → `Mono<ProfileResponse>`
   - `updateProfile(String userId, UpdateProfileRequest)` → `Mono<ProfileResponse>`
   - `changePassword(String userId, String currentPassword, String newPassword, WebSession currentSession)` — BCrypt-verify current; 400 on mismatch; hash new; invalidate all sessions except current via `terminateAllSessionsExcept`; log `password_changed` event
   - `terminateAllSessions(String userId)` — delete all session docs for user via `ReactiveMongoTemplate`
   - `terminateAllSessionsExcept(String userId, String currentSessionId)` — delete all except current session
   - `deleteAccount(String userId, WebSession session)` — set status=deleted, deletedAt=now; invalidate session; log `account_deleted` event

3. **Create profile DTOs**:
   - `UpdateProfileRequest` — `@NotBlank String name` only; `@JsonIgnoreProperties(ignoreUnknown=true)` to silently discard `isSuperAdmin` and any unknown fields
   - `ChangePasswordRequest` — `@NotBlank String currentPassword`, `@NotBlank @ValidPassword String newPassword`
   - `ProfileResponse` — `String id, email, name, status`

4. **Create `SuperAdminSeeder`** at `backend/src/main/java/com/botfunnel/admin/SuperAdminSeeder.java`:
   - `@Component` implementing `ApplicationRunner`
   - `@Value("${app.super-admin.email:}") String adminEmail`, `@Value("${app.super-admin.password:}") String adminPassword`
   - If either blank → log warning and return
   - Find user by email: if `isSuperAdmin=true` → skip; if found but not superadmin → set `isSuperAdmin=true`, save; if not found → create with `status=active`, `isSuperAdmin=true`, BCrypt-hashed password
   - Entire body in try/catch(Exception) → log error only, never throw

5. **Create `HardDeleteJob`** at `backend/src/main/java/com/botfunnel/jobs/HardDeleteJob.java`:
   - `@Component`; inject `UserRepository`
   - Annotate method with `@Recurring(id = "hard-delete-users", cron = "0 3 * * *")` and `@Job(name = "Hard delete soft-deleted users")`
   - Query: `findByStatusAndDeletedAtBefore(UserStatus.deleted, Instant.now().minus(30, DAYS))`
   - Collect IDs → log all user IDs and total count → call `userRepository.deleteAll(users)`

**Dependencies:**
Task 5 (UserRepository, UserService complete), Task 6 (AuthService.terminateAllSessions — read field path before implementing ProfileService version), Task 4 (EventService for logging).

**Edge cases:**
- `PATCH /api/profile` unknown JSON fields: `@JsonIgnoreProperties(ignoreUnknown=true)` handles silently
- `SuperAdminSeeder`: BCrypt-hash password before saving — never store plaintext
- `HardDeleteJob`: if no users to delete → log "0 users removed", return normally
- `changePassword`: new password equal to current → allowed (spec does not prohibit it)

**Implementation hints:**
- `terminateAllSessionsExcept`: verify the actual field path in the `sessions` MongoDB collection by inspecting a live document (e.g., via MongoDB Compass or `db.sessions.findOne()`). Common paths are `principal` or `principalName` depending on the Spring Session version.
- In `ApplicationRunner.run()`, `.block()` is acceptable since it runs at startup, not in reactive context
- `@Recurring` annotation is from `org.jobrunr.scheduling.annotations`; `@Job` annotation is from `org.jobrunr.jobs.annotations` — they live in different packages
- The `@Recurring` cron `"0 3 * * *"` is 5-field Unix cron (minute=0, hour=3, daily). Verify this format is supported by the resolved jobrunr-spring-boot-3-starter version. Some versions require 6-field cron (with seconds field: `"0 0 3 * * *"`). Check `@Recurring` annotation javadoc after adding the dependency.
