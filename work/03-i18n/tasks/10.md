# Task 10: App pages translation (dashboard + profile)

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Локалізувати дві app-сторінки (`pages/dashboard.vue` і `pages/profile.vue`) — це найбільший і найскладніший за UI скоуп фічі i18n після auth-форм. Сторінки приховані за middleware (доступні лише залогіненим), тому покривають увесь "усередині продукту" UX, який користувач бачить після входу.

`dashboard.vue` — короткий: pending-banner із посиланням на профіль, welcome-заголовок з умовним ім'ям через interpolation, опис. `profile.vue` — найбільший файл фічі (~480 LOC, 5 секцій): особисті дані з manual validation на ім'я, resend-verification з cooldown-інтерполяцією, change-password з Zod-схемою (валідації мають перебудовуватись при зміні локалі — Decision 11), terminate-all-sessions, delete-account з модалкою. Усі catch-блоки HTTP-помилок мають піти через `useApiError()` (Decision 10) — централізовано замість inline-маппінгу status→текст.

Це Wave 4 паралельно з Task 8 (app.vue + layouts) і Task 9 (auth pages). Залежить від Task 4 (словники з готовими ключами `dashboard.*`, `profile.*`, `errors.*`) і Task 5 (`useApiError` factory). Після цієї задачі весь user-facing UI фічі готовий — Task 11 (Playwright E2E) йде у Wave 5 і покладається на ці переклади у assertions.

Anchored до AC11/AC12/AC13 з tech-spec (`<NuxtLink>` → `<NuxtLinkLocale>`, `navigateTo()` обгорнуто `localePath()`; Zod-схема у `computed()`; API-помилки через `useApiError`) і Decisions 10, 11, 16.

## What to do

1. **dashboard.vue** — повний переклад:
   - Pending-banner текст `Підтвердіть email для повного доступу.` → `t('dashboard.pendingBanner')`.
   - Лінк-текст `Перейти до профілю →` → `t('dashboard.toProfileLink')` (стрілку лишити у словнику або у шаблоні — на розсуд агента, головне щоб обидві локалі мали ту саму візуальну форму).
   - `<NuxtLink to="/profile">` → `<NuxtLinkLocale to="/profile">`.
   - H1 з conditional: `Ласкаво просимо{name ? ', ' + name : ''}` — підхід через дві ключі (`dashboard.welcomeWithName` з `{name}` плейсхолдером і `dashboard.welcomeAnon`); у шаблоні тернарка вибирає ключ. Обидві локалі мають по дві сутності.
   - Параграф `Це ваш дашборд. ...` → `t('dashboard.description')`.

2. **profile.vue** — повний переклад усіх 5 секцій + модалки + error-маппінгів:
   - **Section 1 (Profile info):** заголовки/labels `Профіль`, `Особисті дані`, `Email`, `Ім'я`, `Редагувати`, `Ім'я збережено`, кнопки `Збереження… / Зберегти`, `Скасувати` → `t('profile.*')`.
   - Manual-validation повідомлення (`Ім'я обов'язкове`, `Ім'я не може бути довшим за 100 символів`) → `t('profile.name.errors.*')`.
   - Catch-блок `saveName` (generic) — замість inline-літералу використати `useApiError()` factory з contextKey `'profile.saveName'`.
   - **Section 2 (Resend verification):** заголовок `Підтвердження email`, `Ваш email не підтверджено`, `Якщо акаунт існує, ви отримаєте лист найближчим часом.`, `Не вдалося надіслати лист. Спробуйте пізніше` — усе через `t()`. Cooldown-кнопка `Зачекайте {n} с / Надіслати лист повторно` — interpolation `t('profile.resend.cooldown', { seconds: resendCooldown })` + `t('profile.resend.button')`. Параграф `Почекайте 60 секунд` (фіксований у поточному коді) — як `t('profile.resend.waitNotice')` без interpolation.
   - **Section 3 (Change password):** Zod-схему обгорнути у `computed(() => z.object({...}))` per Decision 11; усі повідомлення (`Введіть поточний пароль`, `Мін. 8 символів`, `Має містити хоча б одну літеру`, `Має містити хоча б одну цифру`, `Паролі не співпадають`) через `t()`. `toTypedSchema(unref(schemaComputed))` для `useForm.validationSchema`. Catch-блок (400/429/generic) — замінити трьохгілчасту if/else на `useApiError()` з contextKey `'profile.changePassword'` — словник матиме `errors.profile.changePassword.400`, `.429`, `.generic` per Decision 10. Labels (`Поточний пароль`, `Новий пароль`, `Підтвердження нового пароля`), статуси (`Пароль змінено`, `Зачекайте… / Змінити пароль`) — через `t()`.
   - **Section 4 (Sessions):** `Сесії`, опис, кнопка `Завершення… / Завершити всі сесії` → `t()`. Catch-блок — `useApiError()` з contextKey `'profile.terminateSessions'`. `navigateTo('/auth/login')` обгорнути у `localePath('/auth/login')`.
   - **Section 5 (Delete account):** `Видалення акаунта`, опис, кнопка `Видалити акаунт`, модалка (`Видалити акаунт?`, `Це видалить:`, list items `всі ваші проекти` / `ботів` / `підписників`, `Дію не можна скасувати.`, кнопки `Скасувати` / `Видалення… / Видалити акаунт`) — усе через `t()`. Catch-блок — `useApiError()` з contextKey `'profile.deleteAccount'`. `navigateTo('/auth/login')` обгорнути у `localePath()`.
   - Інші посилання `<NuxtLink>` (якщо з'являться) — `<NuxtLinkLocale>`. У поточному `profile.vue` `<NuxtLink>` немає, але `navigateTo()` зустрічається двічі (terminate-sessions, delete-account) — обидва мають піти через `localePath()`.

3. **Перевірити що жодного hardcoded UA-літерала ≥3 символів не лишилось у двох файлах** (AC12 на feature-level — нуль hardcoded рядків поза `t()`).

4. **Не змінювати** `tests/pages/dashboard.spec.ts` і `tests/pages/profile.spec.ts` — UA-літерали в asserts проходять через дефолтну локаль `uk` (Decision 16: existing tests лишаються; виняток лише `auth.global.spec.ts`). Якщо тест почне падати через зміну DOM-структури або data-test атрибутів — перш ніж правити тести, перечитати Decision 16 і вивірити, що зміни не торкнулись селекторів/розмітки.

## TDD Anchor

Окремих unit-тестів для цієї задачі **не пишемо** — Decision 16 явно фіксує, що існуючі spec-файли (`tests/pages/dashboard.spec.ts`, `tests/pages/profile.spec.ts`) використовуються as-is через дефолтну локаль `uk`, а UA-літерали в assertions ловлять регрес. Smoke-перевірка нижче — це і є TDD-loop: тести запускаються до зміни (sanity baseline), потім після зміни (мають усі пройти без модифікацій).

## Details

**Files:**

- `frontend/pages/dashboard.vue`:
  - **Поточний стан:** 32 LOC, повністю UA. 4 user-facing рядки (banner, link, welcome, description). 1 `<NuxtLink to="/profile">`. `<script setup>` тривіальний — лише `useAuthStore()`.
  - **Що зробити:** додати `const { t } = useI18n()`, замінити 4 рядки на `t()`, замінити `<NuxtLink>` на `<NuxtLinkLocale>`, виразити conditional welcome через дві ключі (`welcomeWithName` / `welcomeAnon`).

- `frontend/pages/profile.vue`:
  - **Поточний стан:** ~480 LOC. `<script setup>`: 5 функцій-обробників + 1 Zod-схема (eagerly built на module-scope, рядки 106–119), 5 catch-блоків з inline status→text маппінгом (рядки 56–59, 93–102, 152–163, 179–182, 208–211). Шаблон: 5 `<section>` + 1 модалка, ~30+ UA-літералів. 2 `navigateTo('/auth/login')` (рядки 178, 207). Жодного `<NuxtLink>` поки що.
  - **Що зробити:** додати `useI18n()` + `useLocalePath()` + `useApiError()` у `<script setup>`. Перенести Zod-схему в `computed()` (Decision 11) — увага на правильне `unref()` при передачі у `toTypedSchema`. Замінити inline-error-маппінги на `useApiError()` виклики у кожному catch. У шаблоні всі літерали → `t()`, інтерполяції — через named-параметри (`{ seconds: resendCooldown }`). Обгорнути обидва `navigateTo('/auth/login')` у `localePath('/auth/login')`.

**Dependencies:**
- Task 4 (словники uk.json / en.json) — словник має містити ключі `dashboard.*`, `profile.*`, `errors.profile.*` для кожного contextKey з useApiError.
- Task 5 (`useApiError` factory composable).
- Wave 4 паралель: Task 8 (app.vue / layouts) і Task 9 (auth pages) — не торкаються profile / dashboard, конфліктів немає.

**Edge cases:**
- Welcome-заголовок з conditional ім'ям — два ключі (`welcomeWithName` / `welcomeAnon`) чистіше за одну ключу з пустим `name=''`, бо не залежить від пунктуації локалі.
- Cooldown-параграф "Почекайте 60 секунд" у resend-секції зараз hardcode `60`. Перенести як простий `t('profile.resend.waitNotice')` без інтерполяції — UI поведінка не змінюється.
- Zod-схема у `computed()` — при зміні локалі `useForm` має побачити нову схему. Vee-validate реактивно перебудовує валідацію через `toTypedSchema(unref(schemaComputed))`; перевірити, що активні помилки оновлюються одразу (для цього user-verify крок із power user-flow перевіряє).
- `useApiError()` має бути викликаний на верхньому рівні `<script setup>` — НЕ всередині catch (Decision 10 — інакше lifecycle violation `useI18n called outside setup`).
- Existing tests мажуть очікувати точні UA-рядки. Дефолтна локаль `uk` зберігається — `t('profile.editButton')` віддасть `'Редагувати'` дослівно. Якщо тест посилається на `data-test` атрибути — селектори не змінюються. Якщо тест шукає текст regex'ом по UA — він теж пройде. Decision 16 покриває цю інваріантність.
- Модальний `aria-labelledby="delete-modal-title"` залишається таким же ID; перекладається лише вміст `<h3>`.

**Implementation hints:**
- Подивитись як Task 9 локалізував auth-сторінки (вже у Wave 4 паралельно — або глянути в коді на момент виконання, або у `code-research.md §1.3`) — pattern `useApiError()` + `computed(() => z.object())` буде однаковий.
- `<NuxtLinkLocale to="/profile">` приймає той самий проп `to` що і `<NuxtLink>` — нічого більше не міняється у markup, тільки тег.
- `useLocalePath()` повертає функцію `(path: string) => string`. Виклик: `await navigateTo(localePath('/auth/login'))` — обгортка одна, тип той самий.
- vue-i18n синтаксис інтерполяції: у шаблоні `{{ t('profile.resend.cooldown', { seconds: resendCooldown }) }}`; у словнику значення `Зачекайте {seconds} с` / `Wait {seconds}s`.
- Не іменувати ключ `delete` у словнику — зарезервоване слово JS у деяких контекстах. Краще `deleteAccount.confirmTitle`, `deleteAccount.confirmCta` тощо.
