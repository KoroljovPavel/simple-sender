# Task 10: App pages translation (dashboard + profile)

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Локалізувати дві app-сторінки (`pages/dashboard.vue` і `pages/profile.vue`) — це найбільший і найскладніший за UI скоуп фічі i18n після auth-форм. Сторінки приховані за middleware (доступні лише залогіненим), тому покривають увесь "усередині продукту" UX, який користувач бачить після входу.

`dashboard.vue` — короткий: pending-banner із посиланням на профіль, welcome-заголовок з умовним ім'ям через interpolation, опис. `profile.vue` — найбільший файл фічі (~480 LOC, 5 секцій): особисті дані з manual validation на ім'я, resend-verification з cooldown-інтерполяцією, change-password з Zod-схемою (валідації мають перебудовуватись при зміні локалі — Decision 11), terminate-all-sessions, delete-account з модалкою. Усі catch-блоки HTTP-помилок мають піти через `useApiError()` (Decision 10) — централізовано замість inline-маппінгу status→текст.

Це Wave 4 паралельно з Task 8 (app.vue + layouts) і Task 9 (auth pages). Залежить від Task 4 (словники з готовими ключами `dashboard.*`, `profile.*`, `errors.*`) і Task 5 (`useApiError` factory). Після цієї задачі весь user-facing UI фічі готовий — Task 11 (Playwright E2E) йде у Wave 5 і покладається на ці переклади у assertions.

Anchored до AC11/AC12/AC13 з tech-spec (`<NuxtLink>` → `<NuxtLinkLocale>`, `navigateTo()` обгорнуто `localePath()`; Zod-схема у `computed()`; API-помилки через `useApiError`) і Decisions 10, 11, 16.

**Заплановані відхилення від tech-spec (зафіксувати у `decisions.md`):**
1. **Decision 11 (виправлення):** передаємо `ComputedRef` напряму у `useForm({ validationSchema: schemaComputed })` без `unref()`. Текст спеки пропонує `toTypedSchema(unref(schemaComputed))` — це баг (snapshot-семантика, ламає live-switch). Vee-validate підтримує реактивні схеми лише через прямий пас ref/computed.
2. **Decision 16 / AC18 (розширення авторизованих винятків):** `tests/pages/dashboard.spec.ts` і `tests/pages/profile.spec.ts` отримують точкову правку — мок `useLocalePath` як identity passthrough. UA-літерали в assertions не змінюються.
3. **Decision 10 (часткове застосування у `onResend`):** 429-гілка лишається імперативною (state-machine + timer); `useApiError()` покриває лише generic-error гілку — factory повертає string, не вміє виражати state transitions.

## What to do

1. **dashboard.vue** — повний переклад:
   - Pending-banner текст `Підтвердіть email для повного доступу.` → `t('dashboard.pendingBanner')`.
   - Лінк-текст `Перейти до профілю →` → `t('dashboard.toProfileLink')` (стрілку лишити у словнику або у шаблоні — на розсуд агента, головне щоб обидві локалі мали ту саму візуальну форму).
   - `<NuxtLink to="/profile">` → `<NuxtLinkLocale to="/profile">`.
   - H1 з conditional: `Ласкаво просимо{name ? ', ' + name : ''}` — підхід через дві ключі (`dashboard.welcomeWithName` з `{name}` плейсхолдером і `dashboard.welcomeAnon`); у шаблоні тернарка вибирає ключ. Обидві локалі мають по дві сутності.
   - Параграф `Це ваш дашборд. ...` → `t('dashboard.description')`.

2. **profile.vue** — повний переклад усіх 5 секцій + модалки + error-маппінгів:
   - **Section 1 (Profile info):** заголовки/labels `Профіль`, `Особисті дані`, `Email`, `Ім'я`, `Редагувати`, `Ім'я збережено`, кнопки `Збереження… / Зберегти`, `Скасувати` → `t('profile.*')`.
   - Manual-validation повідомлення (`Ім'я обов'язкове`, `Ім'я не може бути довшим за 100 символів`) → `t('profile.name.errors.*')`.
   - Catch-блок `saveName` (generic) — замість inline-літералу використати `useApiError()` factory з contextKey `'profile.saveName'`.
   - **Section 2 (Resend verification):** заголовок `Підтвердження email`, `Ваш email не підтверджено`, `Якщо акаунт існує, ви отримаєте лист найближчим часом.`, `Не вдалося надіслати лист. Спробуйте пізніше` — усе через `t()`. Cooldown-кнопка `Зачекайте {n} с / Надіслати лист повторно` — interpolation `t('profile.resend.cooldown', { seconds: resendCooldown })` + `t('profile.resend.button')`. Параграф `Почекайте 60 секунд` (фіксований у поточному коді) — як `t('profile.resend.waitNotice')` без interpolation.
   - **`onResend` catch — частковий перехід на `useApiError`:** 429-гілка лишається імперативною (виставляє `resendStatus = 'cooldown'` + стартує таймер `resendCooldownTimer`) — це state-machine transition, який `useApiError()` не вміє виразити (factory повертає лише string). Generic-error гілка (non-429): замість inline-літерала `'Не вдалося надіслати лист. Спробуйте пізніше'` беремо повідомлення з `useApiError()`-резолвера (contextKey `'profile.resend'`), кладемо у `resendErrorText` і виставляємо `resendStatus = 'error'`. Структура: `if (status === 429) { /* як було, тільки текст із словника через t */ } else { resendStatus = 'error'; resendErrorText = useApiError(err, 'profile.resend') }`.
   - **Section 3 (Change password):** Zod-схему обгорнути у `computed()` per Decision 11; усі повідомлення (`Введіть поточний пароль`, `Мін. 8 символів`, `Має містити хоча б одну літеру`, `Має містити хоча б одну цифру`, `Паролі не співпадають`) через `t()`. **Реактивна передача у `useForm`:** `const schemaComputed = computed(() => toTypedSchema(z.object({ ... }))); useForm({ validationSchema: schemaComputed })` — передаємо саме `ComputedRef`, без `unref()`. Vee-validate сам тримає підписку на computed і перебудовує валідацію при зміні локалі. **НЕ викликати `unref(schemaComputed)`** — це знімок на момент виклику, який ламає live-switch (стара UA-схема залишиться у формі після перемикання EN). Це навмисне відхилення від тексту Decision 11 у `tech-spec.md`: спека пропонує `toTypedSchema(unref(...))`, але це баг — vee-validate підтримує реактивні schemas саме через прямий пас computed-ref. Зафіксувати у `decisions.md`. Catch-блок (400/429/generic) — замінити трьохгілчасту if/else на `useApiError()` з contextKey `'profile.changePassword'` — словник матиме `errors.profile.changePassword.400`, `.429`, `.generic` per Decision 10. Labels (`Поточний пароль`, `Новий пароль`, `Підтвердження нового пароля`), статуси (`Пароль змінено`, `Зачекайте… / Змінити пароль`) — через `t()`.
   - **Section 4 (Sessions):** `Сесії`, опис, кнопка `Завершення… / Завершити всі сесії` → `t()`. Catch-блок — `useApiError()` з contextKey `'profile.terminateSessions'`. `navigateTo('/auth/login')` обгорнути у `localePath('/auth/login')`.
   - **Section 5 (Delete account):** `Видалення акаунта`, опис, кнопка `Видалити акаунт`, модалка (`Видалити акаунт?`, `Це видалить:`, list items `всі ваші проекти` / `ботів` / `підписників`, `Дію не можна скасувати.`, кнопки `Скасувати` / `Видалення… / Видалити акаунт`) — усе через `t()`. Catch-блок — `useApiError()` з contextKey `'profile.deleteAccount'`. `navigateTo('/auth/login')` обгорнути у `localePath()`.
   - Інші посилання `<NuxtLink>` (якщо з'являться) — `<NuxtLinkLocale>`. У поточному `profile.vue` `<NuxtLink>` немає, але `navigateTo()` зустрічається двічі (terminate-sessions, delete-account) — обидва мають піти через `localePath()`.

3. **Перевірити що жодного hardcoded UA-літерала ≥3 символів не лишилось у двох файлах** (AC12 на feature-level — нуль hardcoded рядків поза `t()`).

4. **Тести `tests/pages/dashboard.spec.ts` і `tests/pages/profile.spec.ts` — авторизовані винятки до Decision 16 / AC18 (limited-scope):** UA-літерали у assertions залишаються незмінними (дефолтна локаль `uk`), але одна точкова правка потрібна на кожен файл — мок `useLocalePath` як identity passthrough, бо без нього `localePath('/auth/login')` у `navigateTo()` і `<NuxtLinkLocale>` рендеряться як `undefined` у jsdom-середовищі (composable не auto-mocked).
   - **Patch у обох файлах (мінімальний):** `mockNuxtImport('useLocalePath', () => () => (path: string) => path)` у блоку моків зверху файлу (поряд з існуючими `mockNuxtImport` для `useAuthStore`, `navigateTo`).
   - **Чому identity passthrough:** з дефолтною локаллю `uk` і стратегією `prefix_except_default` `localePath('/auth/login')` → `'/auth/login'` (без префікса) — отже identity-функція точно відтворює production-поведінку для дефолтної локалі. Жодного assert не змінюємо.
   - **Конкретний impact, який рятує мок:**
     - `profile.spec.ts:184,233` — `expect(navigateToMock).toHaveBeenCalledWith('/auth/login')` — після обгортки виклик стає `navigateTo(localePath('/auth/login'))`; з identity-моком аргумент лишається `'/auth/login'` → assert проходить.
     - `dashboard.spec.ts:43` — `banner.find('a[href="/profile"]')` — `<NuxtLinkLocale to="/profile">` у дефолтній локалі `uk` рендериться як `<a href="/profile">` (per `prefix_except_default` — без префікса для дефолту); assert проходить за умови, що i18n-плагін у jsdom правильно ініціалізовано (Task 4 додає `nuxt.config.ts`-конфіг).
   - **Документувати у `decisions.md`:** додати до списку авторизованих винятків AC18 / Decision 16 два нові файли (`dashboard.spec.ts`, `profile.spec.ts`) — scope обмежено лише доданням `mockNuxtImport('useLocalePath', ...)`. Усі інші зміни у тестах заборонені; якщо assert почне падати через щось інше — перш ніж правити тест, перечитати Decision 16 і вивірити, чи зміни торкнулись DOM/селекторів/розмітки.

## TDD Anchor

**Edge cases:**
- Welcome-заголовок з conditional ім'ям — два ключі (`welcomeWithName` / `welcomeAnon`) чистіше за одну ключу з пустим `name=''`, бо не залежить від пунктуації локалі. Передавати `name` як параметр у обох гілках безпечно (vue-i18n ігнорує неіснуючі плейсхолдери у `welcomeAnon`).
- Cooldown-параграф "Почекайте 60 секунд" у resend-секції зараз hardcode `60`. Перенести як простий `t('profile.resend.waitNotice')` без інтерполяції — UI поведінка не змінюється.
- **Critical (Zod reactivity):** НЕ викликати `unref()` на `schemaComputed` при передачі у `useForm`. `unref(schemaComputed)` повертає поточне значення (`toTypedSchema(z.object(...))`) як plain value — vee-validate тоді не має способу побачити майбутні зміни computed. Стара UA-схема "залипає" у формі і всі повідомлення лишаються українськими після зміни локалі. Vee-validate >=4.7 підтримує `validationSchema: Ref | ComputedRef | () => Schema` нативно — пасуй ref. Це навмисне відхилення від тексту Decision 11 у tech-spec (там є помилка); зафіксувати у `decisions.md`.
- `useApiError()` має бути викликаний на верхньому рівні `<script setup>` — НЕ всередині catch (Decision 10 — інакше lifecycle violation `useI18n called outside setup`). Викликаємо один раз: `const apiError = useApiError()`; у catch — `apiError(err, 'profile.changePassword')` (factory повертає функцію з reactivity на t).
- `onResend` 429 → cooldown переходить state-machine, який `useApiError` НЕ виражає (factory повертає лише string). Ця гілка лишається імперативною, текст для повідомлень cooldown — через `t('profile.resend.cooldown', { seconds })`. Якщо у майбутньому буде потреба в централізованому 429-handling — це окремий рефактор поза скоупом цієї задачі.

- **Reactive schema pattern (correct):**
  ```ts
  const { t } = useI18n()
  const schemaComputed = computed(() =>
    toTypedSchema(z.object({
      currentPassword: z.string().min(1, t('profile.password.errors.currentRequired')),
      newPassword: z.string()
        .min(8, t('profile.password.errors.minLength'))
        .regex(/[a-zA-Zа-яА-Я]/, t('profile.password.errors.needsLetter'))
        .regex(/\d/, t('profile.password.errors.needsDigit')),
      confirmPassword: z.string(),
    }).refine(d => d.newPassword === d.confirmPassword, {
      message: t('profile.password.errors.mismatch'),
      path: ['confirmPassword'],
    }))
  )
  const { handleSubmit, errors, defineField } = useForm({ validationSchema: schemaComputed })
  ```
  Зверни увагу: `validationSchema: schemaComputed` (не `unref(schemaComputed)`, не `schemaComputed.value`). Vee-validate сам тримає підписку.
- **Test mock pattern (для обох spec-файлів):**

  ```ts
  mockNuxtImport('useLocalePath', () => () => (path: string) => path)
  ```
  Розмісти поряд з існуючими `mockNuxtImport` для `useAuthStore`/`navigateTo`.
