# Task 11: Playwright E2E i18n spec

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Створити єдиний Playwright E2E-тест `frontend/e2e/i18n.spec.ts`, який покриває golden-path сценарій перемикання локалі: відкриття сторінки логіну на default-локалі (uk, без префікса), клік по `EN` у `LangSwitcher`, перевірка зміни URL, тексту кнопки, `<html lang>` і EN-повідомлення Zod-валідації при невалідному email. Один спека, ~30 LOC. Webserver піднімається через `playwright.config.ts` (auto-spawn з Task 1). Тест залежить від Wave 4 (Tasks 8, 9, 10) — `LangSwitcher` має бути вмонтований у `auth.vue` layout, `pages/auth/login.vue` має бути перекладений (button label через `t()`), Zod-схема має бути у `computed()` для re-run при зміні локалі. Це фінальна гілка тестової піраміди для фічі: разом з unit-тестами (`useApiError`, `LangSwitcher`, `strip-invalid-locale`, auth middleware) E2E підтверджує end-to-end інтеграцію `@nuxtjs/i18n` зі сторінками й компонентами.

## What to do

- Створити файл `frontend/e2e/i18n.spec.ts` з одним `test()`-блоком (Playwright `@playwright/test`).
- Сценарій:
  1. `page.goto('/auth/login')` — default uk, без префікса.
  2. Перевірити URL не містить `/en/` (наприклад `expect(page).toHaveURL(/\/auth\/login$/)`).
  3. Перевірити, що `<html lang>` дорівнює `uk` (через `page.locator('html')` + `getAttribute('lang')`).
  4. Перевірити, що видима submit-кнопка має текст `Увійти` (значення з `auth.login.submit` у `uk.json`).
  5. Клікнути по елементу `EN` у `LangSwitcher`.
  6. Дочекатись навігації; перевірити URL `/en/auth/login`.
  7. Перевірити, що `<html lang>` дорівнює `en`.
  8. Перевірити, що submit-кнопка тепер має текст `Sign in` (значення з `auth.login.submit` у `en.json`).
  9. Заповнити email-поле значенням `invalid` і submit-нути форму (клік по submit-кнопці або blur, в залежності від того як налаштована live-валідація після Task 9).
  10. Перевірити, що відображається EN-валідаційне повідомлення (точний текст з `validation.email` у `en.json` — наприклад `Invalid email format` або відповідне формулювання, обране у Task 4).
- Прочитати `frontend/i18n/locales/en.json` (створений у Task 4) щоб взяти точні рядки для assertions: `auth.login.submit`, `validation.email`. Це критично — assertions мають співпасти з реальними значеннями ключів, інакше тест буде flaky.
- Селектори: text-based locator (`page.getByRole('button', { name: 'Увійти' })`) або `page.getByText('EN', { exact: true })` для `LangSwitcher`. Якщо у Task 6 додано `data-testid` — використати його як стабільний якір.
- Тримати спеку коротким (~30 LOC) — без хелперів, без fixtures, без page-object pattern. Один лінійний сценарій.

## Details

**Files:**
- `frontend/e2e/i18n.spec.ts` — новий файл, ~30 LOC, один `test()`-блок з описаним вище сценарієм.

**Dependencies:**
- Task 1 (Wave 1) — Playwright bootstrap: `@playwright/test`, `playwright.config.ts` з webServer, npm-script `test:e2e`, директорія `e2e/`.
- Task 4 (Wave 3) — `frontend/i18n/locales/en.json` з ключами `auth.login.submit`, `validation.email`.
- Task 6 (Wave 3) — `LangSwitcher.vue` — текстовий toggle, клік-handler на елементі `EN`.
- Task 8 (Wave 4) — `LangSwitcher` вмонтований у `auth.vue` layout (absolute top-right).
- Task 9 (Wave 4) — `pages/auth/login.vue` повністю перекладений: submit-кнопка через `t('auth.login.submit')`, Zod-схема у `computed()` (для re-run при locale change — інакше assertion на EN-validation message буде падати).

**Edge cases:**
- Race-condition між `setLocale('en')` і Vue Router navigation — використати `await page.waitForURL(/\/en\/auth\/login/)` після клуку, не покладатись на `expect(page).toHaveURL` без waiter.
- Hydration delay — перед першими assertions може бути потрібен `page.waitForLoadState('domcontentloaded')` або `'networkidle'`. Перевірити локально; якщо тест нестабільний — додати explicit wait.
- LangSwitcher selector: якщо у Task 6 додано `data-testid="lang-switcher-en"` — використати його. Інакше `page.getByText('EN', { exact: true })` (важливо `exact: true` бо `UK` теж рендериться поряд).
- Email-поле: `page.getByLabel('Email')` — `Email` як слово збігається в обох локалях, тому селектор стабільний.
- Validation message: точний рядок з `en.json` `validation.email` — прочитати файл перед написанням assertion і вставити дослівно. Якщо ключ ще не доданий у Task 4 — узгодити з виконавцем Task 4 та додати.
- `<html lang>` поведінка — `useLocaleHead({ seo: true })` (Task 8) має забезпечити оновлення; якщо assertion падає на крок 7 — діагностувати чи `useLocaleHead` правильно прокинутий через `useHead`.

**Implementation hints:**
- Один import: `import { test, expect } from '@playwright/test';`.
- Не створювати fixtures, helpers, page-objects — спека маленька, додаткова інфраструктура буде overkill.
- Не використовувати `test.skip` / `test.fixme` / retry-logic — тест має бути детермінованим.
- Test name має бути описовим (наприклад `'switches locale on auth/login page'`) — `--list` має його вивести.
- Не залежати від real backend — POST `/api/auth/login` НЕ виконується (форма не submit-иться через Zod validation fail на email).
- Якщо у Task 9 live-validation спрацьовує тільки після submit — клікнути submit-кнопку; якщо на blur — викликати `page.getByLabel('Email').blur()`. Простіше — клік submit, оскільки Zod errors відображаються одразу при submit-attempt.
