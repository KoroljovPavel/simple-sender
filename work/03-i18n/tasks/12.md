# Task 12: Code Audit

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:code-reviewing` — [skills/code-reviewing/SKILL.md](../../../.claude/skills/code-reviewing/SKILL.md)

## Description

Holistic code-quality аудит фічі i18n після всіх implementation waves (Tasks 1–11). Аудитор читає **усі** source-файли, створені/змінені у feature scope, і фіксує порушення Decisions / AC / patterns у структурованому JSON-репорті.

Це Audit Wave-задача — аудитор сам **є** фінальним рев'ювером, тому власних `reviewers` не має. На відміну від implementation-задач, тут не пишемо код і не виконуємо тести: завдання — холістично перевірити, що зрозбіжність із tech-spec'ом не прокралась повз per-task code-review раундів. Per-task review бачить лише локальний diff; cross-cutting проблеми (residual hardcoded strings у одній з 12 vue-сторінок, parity drift між middleware allow-list і routeRules, дубльовані patterns у різних файлах, нерівномірне застосування factory-pattern для `useApiError`) ловляться лише на feature-scope огляді.

Вихід — `work/03-i18n/logs/audit/code-audit.json` зі знайденими findings (severity, файл/рядок, цитата, рекомендація). Якщо findings нема — JSON містить empty `findings: []` і `summary: "OK"`. Findings з severity `blocker`/`high` блокують Pre-deploy QA (Task 15).

## Audit Dimensions

Кожен вимір — пункт чек-листа, по якому аудитор робить targeted scan і документує result. Якщо аудитор знаходить порушення — finding з severity згідно з матрицею нижче.

1. **Residual hardcoded UA/EN strings.** Сканувати `frontend/{pages,layouts,components}/**/*.vue` (включно з `app.vue`) на наявність видимого тексту ≥3 символів поза `t()`/`$t()`-викликами. Allowed exceptions: brand → `t('brand.name')` (literal "Bot Funnel" тільки у `i18n/locales/{uk,en}.json`); технічні значення (CSS-класи, ім'я роутів, key атрибути, data-test-id, ARIA-роль values, числа). Перевірити шаблон, default-slots, attr-binding (`title=`, `placeholder=`, `aria-label=`, `alt=`). Severity: **high** (видимий рядок) або **medium** (atribute без user-visible impact).
2. **`<NuxtLink>` → `<NuxtLinkLocale>` parity.** Перевірити, що **жоден** `<NuxtLink ...>` у feature scope не залишився. Усі — `<NuxtLinkLocale>` (per AC, Decision 6). Severity: **high** (прямий лінк ламає i18n routing).
3. **`navigateTo()` обгорнуто `localePath()`.** У scripts і setup-блоках усіх `.vue`/`.ts` файлів feature scope: кожен `navigateTo('...')` обгорнутий `localePath('...')` — крім випадків, де передається name-based route object (`navigateTo({ name: 'dashboard' })` приймає i18n route names напряму). Severity: **high** (лінк ламається на en).
4. **`useApiError` factory pattern correctness.** `useApiError()` має викликатись у `<script setup>` (поза catch-блоком), повертати handler-функцію, яка вже використовується у catch (per Decision 10). Anti-pattern: `useApiError()` всередині `try { } catch { }` блоку → throw "useI18n outside setup()". Перевірити всі 7 pages (5 auth + dashboard + profile) — wherever catch-блок з API-помилкою існує. Severity: **blocker** (runtime error у production).
5. **`routeRules` parity для `/en/auth/**`.** У `frontend/nuxt.config.ts` ключі `'/auth/**'` і `'/en/auth/**'` мають **ідентичні** headers (`X-Robots-Tag: noindex, nofollow`, `Referrer-Policy: no-referrer`). Перевірити, що `/en/auth/**` згенеровано з `NON_DEFAULT_LOCALES` (не hardcoded — Decision 14, AC10). Severity: **high** (security regression — індексація `?token=`).
6. **`NON_DEFAULT_LOCALES` constant — single source of truth.** Константа використовується **і** у `routeRules`-генерації, **і** у `strip-invalid-locale` middleware allow-list (per Decision 5, AC10). Захист від parity drift при додаванні 3-ї локалі. Hardcoded `['en']` у будь-якому з двох місць → finding. Severity: **medium** (futureproofing, не runtime bug).
7. **Duplicated patterns / DRY violations.** Cross-file duplication: однакові 5+ рядків логіки повторені у ≥2 pages (catch-блоки, validation messages, schema-definition copy-paste). Особливо:
   - Чи всі catch-блоки використовують `useApiError` factory (а не inline `e?.statusCode === 401 ? ... : ...`)?
   - Чи Zod-схеми консистентно у `computed()` (не сирий `z.object()` на module-scope)?
   - Чи cooldown-interpolation patterns однакові у `verify-email` і `profile`?
   Severity: **medium** (maintainability).
8. **Conformance to Decisions.** Targeted spot-checks для кожного Decision 1–17:
   - D1: `@nuxtjs/i18n@^9` у `package.json:dependencies`, не голий `vue-i18n`.
   - D2: `strategy: 'prefix_except_default'` у `nuxt.config.ts:i18n`.
   - D3: `cookieKey: 'i18n_lang'` explicit; `cookieSecure: process.env.NODE_ENV === 'production'`; `cookieMaxAge` НЕ передано (module default).
   - D4: `redirectOn: 'no prefix'`, `alwaysRedirect: false`, `fallbackLocale: 'uk'`.
   - D5: Path normalization order у `strip-invalid-locale.ts` правильний: backslash reject → strip prefix → collapse slashes → decode percent. Auth-zone match `startsWith('/auth')` (без trailing slash).
   - D6: `useRouteBaseName()` + `PUBLIC_ROUTE_NAMES` whitelist + fail-closed на undefined у `auth.global.ts`.
   - D7: `lazy: false` (або відсутність `lazy: true`) у i18n-конфігу.
   - D8: nested JSON namespaces (`auth.*`, `dashboard.*`, `profile.*`, `layout.*`, `validation.*`, `errors.*`, `common.*`, `brand.*`); camelCase keys.
   - D9: `scripts/check-locales.mjs` використовує `Object.create(null)` + фільтр `__proto__`/`constructor`/`prototype`. Vue I18n config: `silentTranslationWarn: true` + `silentFallbackWarn: true` для prod.
   - D10: `useApiError` factory + `te()` fallback chain (`errors.{ctx}.{code}` → `errors.{ctx}.generic` → `errors.generic`).
   - D11: Zod-схеми у `computed()` у всіх формах.
   - D12: LangSwitcher викликає `await setLocale(code)` (не explicit `navigateTo(switchLocalePath(code))`).
   - D13: `brand.name` ключ у обох локалях зі значенням `"Bot Funnel"`.
   - D14: routeRules parity (див. dimension 5).
   - D15: LangSwitcher placement — топбар flex-row у `default.vue`, absolute top-right у `auth.vue`.
   - D16: existing test files (UA-літерали) не модифіковані крім `auth.global.spec.ts`.
   - D17: Wave structure відповідає (Playwright у Wave 1 solo, i18n bootstrap у Wave 2).
   Severity: variable — за впливом конкретного відхилення.
9. **AC compliance spot-check.** Пройтись по 11 tech-spec AC + 19 user-spec AC: чи кожен критерій підтриманий кодом? Особливо AC, які складно перевірити автоматизовано:
   - AC2 (однаковий набір ключів) — перевіряє `check-locales.mjs`, але переконатись що скрипт є у `prebuild`.
   - AC15 (cookie оновлений після setLocale) — перевіряє Vitest `LangSwitcher.spec.ts`.
   - AC18 (existing UA tests не мутовані) — порівняти git history тестових файлів (Decision 16 exception для `auth.global.spec.ts`).
   Severity: за впливом.

## Audit Report Schema

JSON-структура `work/03-i18n/logs/audit/code-audit.json`:

```json
{
  "task": 12,
  "feature": "03-i18n",
  "auditor": "code-reviewing",
  "timestamp": "ISO-8601",
  "summary": "OK | N findings: X blocker, Y high, Z medium, W low",
  "scope_files_count": 0,
  "dimensions_checked": [
    "residual_strings",
    "nuxtlinklocale_parity",
    "navigateto_localepath",
    "useapierror_factory",
    "routerules_parity",
    "non_default_locales_constant",
    "duplicated_patterns",
    "decisions_conformance",
    "ac_compliance"
  ],
  "findings": [
    {
      "id": "F-001",
      "severity": "blocker | high | medium | low",
      "dimension": "<one of dimensions_checked>",
      "file": "frontend/pages/auth/login.vue",
      "line": 42,
      "issue": "Concrete description of what's wrong",
      "quote": "The exact problematic code line(s)",
      "recommendation": "How to fix",
      "anchored_to": "Decision N | AC N | tech-spec section"
    }
  ]
}
```

Severity rubric:
- **blocker** — runtime error / security hole / build failure (наприклад `useApiError` у catch-блоці).
- **high** — порушення AC / Decision з прямим user-visible impact (residual hardcoded string на видимій сторінці, missing `<NuxtLinkLocale>` що ламає en-routing).
- **medium** — maintainability / futureproofing (parity drift, duplicated pattern).
- **low** — nitpick / style (inconsistent comments, JSDoc gaps).

## Details

**Files:**
- `work/03-i18n/logs/audit/code-audit.json` (new) — основний deliverable. Структура per **Audit Report Schema** вище.
- `work/03-i18n/decisions.md` (modify) — додати Summary-entry per Post-completion checklist.

**Dependencies:**
- Task 11 (E2E spec) — completed. Усі implementation waves закриті, код стабільний.
- Skill `code-reviewing` — методологія review (11 dimensions, severity rubric).

**Edge cases:**
- Якщо у Tasks 1–11 був deviation з approved exception у decisions.md — finding має fixate "approved deviation, no action" і не блокувати Pre-deploy QA.
- Якщо файл не існує (наприклад `frontend/scripts/check-locales.test.mjs` — він optional per tech-spec) — це не finding; finding тільки якщо існуючий файл некоректний.
- Якщо аудитор сумнівається у severity — підвищити на 1 рівень (defensive). Краще false-positive high, ніж пропущений real-blocker.
- Якщо `te()` chain у `useApiError` падає на edge case (наприклад `contextKey` undefined) — це **blocker**, бо runtime exception у catch-блоку = double-fault.
- Cross-locale string-leak: якщо у `uk.json` є `"Sign in"` (англійський рядок у українській локалі) — finding **medium** (copywriting issue, не functional bug, але регрес користувацького досвіду).

**Implementation hints:**
- Стартуй з аркушу: для кожної з 9 dimensions виділи окрему секцію в внутрішніх нотатках, перерахуй файли scope, проходь по них один-в-один.
- Для dimension 1 (residual strings) — читай `<template>`-блок як plain text, шукай UA-кирилицю поза `{{ t(...) }}` і EN-літерали з ≥3 символів поза `t()`. Не зараховуй: брендові ім'я "Bot Funnel" у `t('brand.name')`-вимозі (один шлях через ключ); CSS-class names; route names.
- Для dimension 4 (factory pattern) — grep по `useApiError(` у всіх vue-файлах; перевір, що виклик стоїть на module-level setup (поза `try`/`catch`/IIFE).
- Для dimension 5/6 — прочитай `nuxt.config.ts:routeRules` повністю; знайди де визначається `NON_DEFAULT_LOCALES`; перевір що allow-list у `strip-invalid-locale.ts` імпортує саме його.
- Не дублюй per-task review findings — фокус на cross-cutting / holistic. Якщо finding був у task-specific review і там зафіксований як "fixed" — пропусти.
- Якщо JSON виходить великий (>100 findings) — це сигнал що щось систематично не так; зупинись і ескалюй замість великого репорту.
