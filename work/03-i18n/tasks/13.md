# Task 13: Security Audit

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:security-auditor` — [skills/security-auditor/SKILL.md](../../../.claude/skills/security-auditor/SKILL.md)

## Description

Full-feature security audit фічі 03-i18n за стандартом OWASP Top 10. Аудитор читає фінальний стан усіх source-файлів, створених/змінених у попередніх Waves (1–5), і перевіряє вісім фокусних інваріантів безпеки, плюс холістичний OWASP-проход. Фокус — на регресіях, які i18n-інтеграція могла внести у вже захищений auth-сервіс: cookie-flags parity з не-HttpOnly UI-cookie, security headers parity на додатковому glob-патерні `/en/auth/**`, збереження headers на 302-response для auth-zone редіректів (інакше `?token=` витече через Referer проміжної відповіді), open-redirect через path-маніпуляції у server middleware, prototype-pollution безпека build-time JSON-валідатора, та fail-closed semantics auth middleware після переходу на `useRouteBaseName()`.

Аудитор не править код — записує structured report у `work/03-i18n/logs/audit/security-audit.json` зі списком findings (severity, location, issue, recommended fix, anchored Decision/Risk). Findings вищого severity (`high`/`critical`) тригерять fix-cycle на відповідальних tasks; informational/low — фіксуються у decisions.md як known limitations.

## What to do

1. Прочитати фінальний стан усіх source-файлів feature scope (детально нижче у Details). Особливо — `frontend/server/middleware/strip-invalid-locale.ts`, `frontend/middleware/auth.global.ts`, `frontend/scripts/check-locales.mjs`, `frontend/nuxt.config.ts` (cookie config + routeRules), `frontend/i18n.config.ts`.
2. Прочитати tech-spec Decisions 3, 5, 6, 9, 14 і повний Risks-блок — щоб зіставити кожне рішення з його implementation і верифікувати mitigation.
3. Виконати фокусну перевірку 8 інваріантів безпеки (повний перелік — у Acceptance Criteria).
4. Виконати холістичний OWASP Top 10 проход по всьому feature scope: A01 broken access control (auth middleware fail-closed), A02 cryptographic failures (cookie Secure flag), A03 injection (path normalization, JSON parsing), A05 security misconfiguration (headers parity, routeRules), A07 ident/auth (session/locale interaction), A08 software/data integrity (prototype pollution у JSON-вантажі), A10 SSRF/open-redirect (path normalization).
5. Скласти structured report — для кожного finding: `severity` (informational/low/medium/high/critical), `location` (file:line або decision-anchor), `issue` (опис), `fix` (рекомендація), `anchored_to` (Decision/Risk/AC). Якщо інваріант OK — теж зафіксувати pass-line з location-evidence.

## Acceptance Criteria

- [ ] **Cookie-flags parity (Decision 3, AC1):** `i18n_lang` cookie на production має `Secure` флаг (env-conditional `cookieSecure: process.env.NODE_ENV === 'production'`); `SameSite=Lax` (module default); MaxAge ~1 рік (module default); non-HttpOnly justification (UI preference, не auth-token) задокументована і тримається — cookie не використовується для авторизаційних рішень
- [ ] **Security headers parity на `/en/auth/**` (Decision 14, AC8):** і `/auth/**`, і `/en/auth/**` ключі присутні у `routeRules` `nuxt.config.ts` з ідентичними headers (`X-Robots-Tag: noindex, nofollow`, `Referrer-Policy: no-referrer`); patterns generated з єдиного джерела `NON_DEFAULT_LOCALES` (parity-захист per Decision 5)
- [ ] **302 headers preservation у `strip-invalid-locale` (Decision 5, Risk):** для auth-zone редіректів (`/fr/auth/verify-email?token=x` тощо) headers `X-Robots-Tag: noindex, nofollow` і `Referrer-Policy: no-referrer` виставляються НА сам 302-response (не загублені на проміжній відповіді) — інакше `?token=` витече через Referer-header браузера
- [ ] **Open-redirect нормалізація у path (Decision 5, Risk):** path normalization order дотримано — backslash reject (`/fr/\\evil.com` → 400) → strip prefix → collapse leading slashes (`/fr//evil.com` → `/evil.com`, не `//evil.com`) → decode percent-encoding ПІСЛЯ strip+normalization (захист від decode-then-match bypass з double-encoded payloads)
- [ ] **Prototype-pollution safety у `check-locales.mjs` (Decision 9, Risk):** key-set збирається через `Object.create(null)` (no prototype chain); ключі `__proto__`, `constructor`, `prototype` фільтруються; payload `{"__proto__": {...}}` у локаль-JSON не забруднює shared prototype після парсингу (no `Object.assign`-style merge)
- [ ] **`?token=` URLs не leakуються через Referer (Decision 5, AC8, Risk):** на verify-email і reset-password flows у обох локалях (`/auth/verify-email?token=x`, `/en/auth/verify-email?token=x`, та невалідні префікси `/fr/auth/verify-email?token=x`) — `Referrer-Policy: no-referrer` присутній на всіх response paths (200, 302, 4xx)
- [ ] **Fail-closed semantics в auth middleware (Decision 6):** `getBaseName(to)` undefined (route не resolved, наприклад catch-all 404) → middleware treat as protected (redirect на login), не passthrough; whitelist `PUBLIC_ROUTE_NAMES` — explicit closed list, не open regex/startsWith
- [ ] **Production-mode warning suppression (Decision 9):** `silentTranslationWarn: true` і `silentFallbackWarn: true` встановлені у production-mode конфігу `frontend/i18n.config.ts` — захист від info disclosure через console з missing-key paths у production logs
- [ ] **OWASP Top 10 holistic check** виконано — кожна категорія A01–A10 явно перевірена для feature scope, findings (or "no issues") зафіксовані у звіті
- [ ] Report записаний у `work/03-i18n/logs/audit/security-audit.json` зі списком findings (severity + location + issue + fix + anchored_to) і pass-lines для перевірених інваріантів
- [ ] Усі finding'и `high`/`critical` мають конкретний actionable fix (не "review later"); `medium` мають recommendation + acceptance owner; `low`/`informational` мають justification якщо acceptable

**Files (read-only, фінальний стан Wave 5):**
Усі файли feature scope — повний перелік у Context Files. Read-only — аудитор не править код, лише записує findings.

**Dependencies:**

- Залежить від Task 11 (Playwright E2E) — фактично залежить від усього Wave 5 готовності, бо Task 11 — остання implementation-task. Audit Wave запускається паралельно після Wave 5.

**Edge cases для перевірки:**

- Cookie без `Secure` у production (env mismatch)
- routeRules має `/auth/**` але нема `/en/auth/**` (parity drift)
- 302 headers ставляться на final-response, не на 302 (browser отримує `Referer` із `?token=`)
- Path normalization: `/fr/%2F%2Fevil.com` (URL-encoded double slash) — decode-then-match bypass якщо decode перед normalization
- Path normalization: `/fr/\evil.com` vs `/fr/\\evil.com` — рівне ставлення до single і double backslash
- `check-locales.mjs` парсить `{"__proto__": {"polluted": true}}` — після парсингу `({}).polluted` має бути `undefined`
- `auth.global.ts` для route name `undefined` (catch-all 404 page) — fail-closed
- `silentTranslationWarn` встановлено лише в development (інверсія) — info disclosure у prod
- hreflang для auth-сторінок не leakує `?token=` через `<link rel="alternate">` теги (мають базуватися на canonical path без query)

**Implementation hints (для аудитора):**

- Робота — read-only review, не write/edit. Findings — у JSON-output, не у inline-коментарях.
- Severity criteria: `critical` — реальний exploitable vector (наприклад token leak, open-redirect); `high` — high-impact regression (наприклад parity drift, missed Secure flag у prod); `medium` — defense-in-depth gap (наприклад missing fail-closed branch); `low` — code-hygiene security (наприклад comment outdated); `informational` — observation без impact.
- Кожен finding anchored до Decision/Risk/AC — щоб fix-cycle мав чіткий контекст.
- Якщо audit виявляє відсутність invariant у коді (наприклад, `Object.create(null)` не використано) — це finding `high`, не "skip if missing".
- Pass-lines обов'язкові для всіх 8 фокусних інваріантів — навіть якщо нема findings, треба зафіксувати evidence (file:line).
- При перевірці `check-locales.mjs` — імітувати payload-парсинг: чи `JSON.parse('{"__proto__": {...}}')` забруднює `Object.prototype`? У Node.js JSON.parse безпечний за дефолтом для `__proto__` як own-property — але `Object.assign`/spread на result може shift це у prototype chain. Перевірити логіку traversal.
