## Description

Full-feature тестовий аудит для фічі `03-i18n`. Виконується після всіх implementation-хвиль (Wave 1-5) як частина Audit Wave паралельно з Code Audit (Task 12) і Security Audit (Task 13).

Задача — холістично прочитати всі test-файли feature scope і перевірити їх якість як одного цілого: чи виконують вони реальну роботу (meaningful assertions), чи правильно розподілені по піраміді (M-feature: 4 нові unit + 1 оновлений middleware spec + 1 E2E), чи покривають усі критичні шляхи з Testing Strategy техспека (рядки 192-208), чи дотримано AC18 (existing UA-specs не мутовані) з єдиним authorized exception на `auth.global.spec.ts` (per Decision 16), чи чисто з mock-гігієною (`mockNuxtImport` де треба, без реальних мережевих викликів), і чи присутній prototype-pollution фікстур у `check-locales.test.mjs`.

Аудит read-only: жодних правок у test-файли. Знахідки записуються у `work/03-i18n/logs/audit/test-audit.json` для подальшого розгляду на Pre-deploy QA (Task 15) і командою.

## What to do

1. Прочитати усі test-файли фічі (повний список у Context Files): 4 нові Vitest specs, 1 оновлений middleware spec, 1 E2E spec, 1 опційний node:test файл для check-locales.
2. Прочитати Testing Strategy техспеку (рядки 192-208) як reference для критичних шляхів.
3. Прочитати Decision 16 (рядки 160-164 техспеку) для розуміння AC18 exception scope.
4. Прочитати tech-spec Acceptance Criteria рядок 265 ("9 існуючих + 4 нові: useApiError, LangSwitcher, strip-invalid-locale, check-locales; auth.global.spec.ts оновлений") як baseline для pyramid-balance перевірки.
5. Для кожного test-файлу оцінити шість аудит-вимірів:
   - **Meaningful assertions** — кожен тест підтверджує observable behavior (call зі spy + аргумент, повернене значення, side-effect на cookie/store), а не лише "функція існує" або "не кинула виключення".
   - **Pyramid balance** — реальна структура (4 unit + 1 updated middleware + 1 E2E) відповідає M-feature розміру; жодного зайвого/відсутнього шару.
   - **Coverage критичних шляхів** — усі test-cases перелічені у Testing Strategy техспеку присутні (fallback chain у useApiError; cookie value + setLocale spy у LangSwitcher; всі decision-логіки strip-invalid-locale; всі нові кейси у auth.global.spec.ts; key-divergence + prototype-pollution у check-locales; URL+text+`<html lang>` у E2E).
   - **AC18 compliance** — перевірити dual-scope:
     - **Authorized exceptions (8 файлів, per Decision 16 + Tasks 9, 10):** `tests/middleware/auth.global.spec.ts`, `tests/pages/auth/login.spec.ts`, `tests/pages/auth/register.spec.ts`, `tests/pages/auth/reset-password.spec.ts`, `tests/pages/auth/forgot-password.spec.ts`, `tests/pages/auth/verify-email.spec.ts`, `tests/pages/dashboard.spec.ts`, `tests/pages/profile.spec.ts`. Дозволений scope мутації для цих 8 файлів обмежений: (a) додавання `mockNuxtImport('useLocalePath', () => () => (path) => path)` (identity passthrough, що зберігає uk-default behavior); (b) для `verify-email.spec.ts` додатково — перейменування `t`-collision-references якщо вони є; (c) **UA-літерали в assertions ОБОВ'ЯЗКОВО залишаються незмінними**. Будь-яка зміна поза цим scope (новий `vi.mock('vue-i18n')`, `t()`-mock, заміна UA-strings на `t()`-keys, нові import statements не для `useLocalePath` mock) = AC18 violation.
     - **Решта existing UA-specs (поза 8-ма authorized exceptions):** жодних мутацій взагалі — UA-літерали типу `/коректний email/i` залишилися, жодного нового mock, жодних змін у assertion strings.
   - **Mock hygiene** — `mockNuxtImport` використано там де треба (`useI18n`, `useRouteBaseName`, `useLocalePath`, `useCookie`); жодних реальних HTTP-викликів (vitest unit-шар), Playwright-only робить мережу.
   - **Prototype-pollution fixture** — `check-locales.test.mjs` має кейс із `__proto__`-payload, що підтверджує безпечність парсингу (per Decision 9).
6. Зібрати знахідки у структурований JSON у форматі `{file, dimension, severity, issue, recommendation}`.
7. Записати фінальний звіт у `work/03-i18n/logs/audit/test-audit.json` з полями: `feature`, `task`, `auditor`, `date`, `summary`, `findings[]`, `pyramid_breakdown` (counts по шарах), `coverage_check` (mapping testing-strategy кейсів на знайдені тести), `ac18_compliance` (boolean + notes), `mock_hygiene` (boolean + notes), `prototype_pollution_fixture_present` (boolean), `verdict` (`pass` / `pass-with-recommendations` / `fail`).
8. Якщо знайдено `severity: high` — підсвітити у summary і верхньому-рівневому полі `blocking_findings: true`. Pre-deploy QA (Task 15) не запускає suite поки blocking не resolved.

**Dependencies:**
- Task 11 (E2E spec) — фінальний implementation task, після нього усі test-артефакти присутні
- Tech-spec Testing Strategy + Decision 16 — обов'язкові для розуміння expected pyramid та AC18 exception

**Edge cases:**
- Якщо `check-locales.test.mjs` не створений (опційний у Task 4) — записати у findings як `informational`, не `high`. Pyramid breakdown все ще валідний: 4 unit + 1 updated middleware + 1 E2E.
- Якщо знайдено мутацію existing UA-spec поза 8-ма authorized exceptions (`auth.global.spec.ts` + 7 page specs: `tests/pages/auth/{login,register,reset-password,forgot-password,verify-email}.spec.ts`, `tests/pages/dashboard.spec.ts`, `tests/pages/profile.spec.ts`) — це `severity: high` (порушення AC18 + Decision 16) і `blocking_findings: true`.
- Якщо у будь-якому з 8 authorized exceptions знайдено зміни поза дозволеним scope — нові `vi.mock('vue-i18n')`, `t()`-mock замість `useLocalePath` mock, заміна UA-літералів у assertions на `t()`-keys, інші imports не пов'язані з `useLocalePath` mock — це теж `severity: high` (out-of-scope mutation, порушення AC18).
- Якщо `LangSwitcher.spec.ts` не асертить cookie value (per AC15 user-spec вимагає "cookie оновлений") — `severity: high`, бо AC15 explicit.
- Якщо в `auth.global.spec.ts` відсутні нові кейси для en-локалі (`useLocalePath` повертає `/en/auth/login`) — `severity: high`, бо Testing Strategy явно вимагає coverage обох локалей.
- Якщо у `useApiError.spec.ts` тести лише підтверджують factory-pattern без асертів на fallback chain (`errors.{context}.{code}` → `errors.{context}.generic` → `errors.generic`) — `severity: medium`, бо governing decision (Decision 10) — це fallback gradient.

**Implementation hints:**
- Аудит не запускає тести — це read-only review артефактів. Запуск suite — задача Task 15 (Pre-deploy QA).
- Diff проти baseline для AC18 compliance: можна порівняти з `git log --all --oneline -- frontend/tests/pages/ frontend/tests/middleware/` або просто читати existing specs і шукати індикатори out-of-scope мутації — `t()`-mock, `vi.mock('vue-i18n')`, заміна UA-літералів у assertions на `t()`-keys. Для 8 authorized exceptions очікуваний diff обмежений `mockNuxtImport('useLocalePath', () => () => (path) => path)` (плюс опційне перейменування `t`-collisions у `verify-email.spec.ts`); будь-що інше = violation.
- Mock hygiene: `mockNuxtImport` — це Nuxt-специфічний spelling замість стандартного `vi.mock('#imports')`. Шукати імпорт `import { mockNuxtImport } from '@nuxt/test-utils/runtime'` у specs що потребують auto-imports.
- Pyramid breakdown — коротка таблиця `{layer: count}`: `{unit: 4, integration: 0, e2e: 1, helper-tests: 1}`. Integration намірено 0 (фіча 100% frontend, per Testing Strategy line 204).
- Severity rubric:
  - `high` — порушення AC, Decision або canonical pattern (cookie not asserted, AC18 violated, prototype-pollution fixture відсутній)
  - `medium` — coverage gap у non-critical path (fallback gradient частково покрито, edge case не передбачено)
  - `low` — стилістика, naming, дублювання, що не впливає на correctness
  - `informational` — опційні фічі не реалізовані (наприклад, опційний check-locales.test.mjs)
