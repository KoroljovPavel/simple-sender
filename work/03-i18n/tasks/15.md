# Task 15: Pre-deploy QA

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:pre-deploy-qa` — [skills/pre-deploy-qa/SKILL.md](.claude/skills/pre-deploy-qa/SKILL.md)

## Description

Фінальне приймальне тестування фічі `03-i18n` перед деплоєм. Задача — переконатися, що вся реалізація i18n (Waves 1–5) і audit-результати (Wave 6: Tasks 12–14) дають working state, що відповідає user-spec (19 acceptance criteria) і tech-spec (11 додаткових технічних AC). Задача також враховує знахідки з audit-репортів (`logs/audit/code-audit.json`, `security-audit.json`, `test-audit.json`) — якщо аудитори знайшли blockers, QA фіксує статус і блокує deploy.

QA виконується власноруч агентом — без code-review/security-audit/test-review reviewers (audit-фаза вже відбулася у Wave 6, QA є власною верифікацією). На виході — JSON-репорт у `work/03-i18n/logs/qa/pre-deploy-qa.json` зі статусом per-AC, виконаними командами, observed output і фінальним verdict (PASS / PASS_WITH_ISSUES / BLOCK_DEPLOY).

Скоуп QA: автоматичні тести (Vitest unit/component + Playwright E2E), curl-смок-перевірки 8 сценаріїв з user-spec "Як перевірити", build-fail тест (видалити `en.json` → `pnpm build` падає), manual cookie inspection через DevTools для AC3–AC6 (різні стани cookie + Accept-Language), параметри cookie (Secure у production-build, SameSite Lax, MaxAge ~1 рік). Deploy і post-deploy verification — out of scope (CI/CD не існує per `deployment.md`, фіча — frontend-bundle only).

## What to do

1. Прочитати `work/03-i18n/user-spec.md` (19 AC + секція "Як перевірити") і `work/03-i18n/tech-spec.md` (Acceptance Criteria — 11 додаткових технічних AC, lines 259–271, та Agent Verification Plan).
2. Прочитати audit-репорти у `work/03-i18n/logs/audit/`: `code-audit.json`, `security-audit.json`, `test-audit.json`. Зафіксувати у QA-репорті всі blocker/critical-знахідки і чи були вони fixed (cross-reference з commits після audit Wave).
3. Запустити повний Vitest suite: `cd frontend && pnpm test` — усі specs зелені (мінімум 9 існуючих + 4 нові: useApiError, LangSwitcher, strip-invalid-locale, check-locales; auth.global.spec.ts оновлений).
4. Запустити Playwright E2E: `cd frontend && pnpm test:e2e` — `e2e/i18n.spec.ts` проходить (open `/auth/login` → click EN → URL `/en/auth/login` + кнопка "Sign in" + EN-валідація + `<html lang>="en"`).
5. Запустити dev-сервер `cd frontend && pnpm dev` і виконати 8 curl-сценаріїв з user-spec "Як перевірити":
   - `curl -sI 'http://localhost:3000/en/auth/verify-email?token=x'` → headers містять `X-Robots-Tag: noindex, nofollow` + `Referrer-Policy: no-referrer`.
   - `curl -sI -H 'Accept-Language: en-US' http://localhost:3000/dashboard` → 302 на `/en/...`, Set-Cookie `i18n_lang=en`.
   - `curl -sI -H 'Accept-Language: uk-UA' http://localhost:3000/dashboard` → 302 на `/auth/login` (без префікса), cookie `uk`.
   - `curl -sI http://localhost:3000/fr/dashboard` → 302 location `/dashboard`.
   - `curl -sI http://localhost:3000/uk/dashboard` → 302 location `/dashboard` (стрипається default-locale prefix).
   - `curl -sI -H 'Cookie: i18n_lang=xx' http://localhost:3000/dashboard` → невалідне cookie ігнорується, fallback на Accept-Language → default uk (AC6).
   - `curl -sI 'http://localhost:3000/fr/auth/verify-email?token=x'` → 302 на `/auth/verify-email?token=x` + `X-Robots-Tag` + `Referrer-Policy` headers ВЖЕ на 302-response (не лише на target).
   - `curl -sI 'http://localhost:3000/auth/verify-email?token=x'` → headers `X-Robots-Tag` + `Referrer-Policy` (parity-check `/auth/**` vs `/en/auth/**`).
6. Build-fail тест: `cd frontend && cp i18n/locales/en.json /tmp/en-backup.json && rm i18n/locales/en.json && pnpm build` — build падає на prebuild script з зрозумілим повідомленням (exit code != 0). Відновити: `cp /tmp/en-backup.json frontend/i18n/locales/en.json`. Підтвердити, що build знову проходить.
7. Production-build cookie attribute parity: `cd frontend && pnpm build && pnpm preview` (або еквівалентний production-mode dev), потім `curl -sI -H 'Accept-Language: en-US' http://localhost:3000/dashboard` → Set-Cookie `i18n_lang=en` має `Secure`, `SameSite=Lax`, `Max-Age` ~31536000 (1 рік ±tolerance per Decision 3, бо module default).
8. Manual DevTools перевірки (агент описує кроки в репорті, виконує власноруч через скриптовані curl-запити з explicit Cookie/Accept-Language headers — еквівалент DevTools для server-side detection):
   - AC3: очистити cookies, відкрити `/dashboard` з `Accept-Language: en-US` → редирект на `/en/...`, cookie `i18n_lang=en` встановлений.
   - AC4: очистити cookies, відкрити `/dashboard` з `Accept-Language: uk-UA` → uk без префікса, cookie `uk`.
   - AC5: cookie `i18n_lang=en`, відкрити `/dashboard` (без Accept-Language en) → 302 на `/en/dashboard`.
   - AC6: cookie `i18n_lang=xx` (поза списком) → cookie ігнорується, fallback працює.
9. Walkthrough всіх 19 AC з user-spec + 11 AC з tech-spec — для кожного зафіксувати: метод перевірки (test name / curl / manual), observed output, статус (PASS/FAIL/BLOCKED).
10. Cross-check audit-blockers: якщо audit-репорти містять unresolved blocker/critical → QA verdict = BLOCK_DEPLOY з посиланням на конкретні finding ids.
11. Записати фінальний QA-репорт у `work/03-i18n/logs/qa/pre-deploy-qa.json` зі структурою (приблизно):
    ```
    {
      "feature": "03-i18n",

      "verdict": "PASS | PASS_WITH_ISSUES | BLOCK_DEPLOY",
      "automated_tests": { "vitest": {...}, "playwright": {...} },
      "smoke_checks": [ {"scenario": "...", "command": "...", "expected": "...", "observed": "...", "status": "PASS|FAIL"}, ... ],

      "tech_spec_ac": [ ...11 items ],
      "audit_blockers_resolved": [...],
      "deferred_to_post_deploy": [],
    ```

## Details

**Files:**
- `work/03-i18n/logs/qa/pre-deploy-qa.json` (new) — фінальний QA-репорт зі структурою описаною у `What to do` step 11. Створити директорію `logs/qa/` якщо її ще нема (`mkdir -p work/03-i18n/logs/qa`).

**Dependencies:**
- Task 12 (Code Audit) — `logs/audit/code-audit.json` має існувати.
- Task 13 (Security Audit) — `logs/audit/security-audit.json` має існувати.
- Task 14 (Test Audit) — `logs/audit/test-audit.json` має існувати.
- Усі попередні Waves (1–5) — implementation має бути закінчена і committed.

**Edge cases:**
- Audit-репорти можуть бути відсутні (Wave 6 не запустилася) — у цьому разі QA фіксує це як blocker.
- Dev-сервер може не стартувати на чистому checkout — підтвердити `pnpm install` і `pnpm dev` перед running curl-сценаріїв.
- Production-build cookie test може не дати Secure-флаг якщо тестується на HTTP localhost — це expected per Decision 3 (env-conditional). Перевірка `Secure` робиться у production-build режимі (NODE_ENV=production), не у dev. Якщо середовище не дозволяє істинний production-mode локально — зафіксувати у репорті як deferred-to-post-deploy.
- `pnpm preview` потребує prior `pnpm build` — впевнитись що build success перед preview.
- Manual DevTools перевірки можуть бути виконані headlessly через scripted curl з explicit Cookie/Accept-Language headers — це еквівалентно DevTools для server-side detection.
- Tech-spec User-Spec Deviations містить APPROVED-deviation на cookie TTL (module default ~1y замість explicit) — не помилково треба flag-увати як failure.

**Implementation hints:**
- Запустити `pnpm dev` у фоні (`run_in_background`), почекати ~5–10 секунд на старт, потім виконувати curl. У кінці зупинити process.
- Для парсингу Set-Cookie і headers — використати grep або `awk` на curl output.
- Audit-blockers cross-check: прочитати кожен audit JSON, фільтрувати по `severity: critical|blocker`, перевірити чи finding fixed (можна шукати в git log по date+keyword).
- Verdict = PASS лише якщо всі AC PASS + жодного unresolved audit-blocker. Інакше PASS_WITH_ISSUES (cosmetic/minor) або BLOCK_DEPLOY (critical).
- Per tech-spec footer (line 400): Deploy + Post-deploy verification out of scope; не запускати DEPLOY-related checks (немає env, немає prod URL).
