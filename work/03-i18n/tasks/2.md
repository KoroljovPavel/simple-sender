# Task 2: i18n module bootstrap

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [.claude/skills/code-writing/SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Підключаємо офіційний `@nuxtjs/i18n@^9` до Nuxt 4 фронтенду — це foundation-задача всієї фічі i18n: усі подальші waves (server middleware, локаль-словники, composables, компоненти, переклад сторінок, E2E) залежать від коректно зареєстрованого модуля з правильною cookie-стратегією, locale-strategy `prefix_except_default` (uk без префікса, `/en` з префіксом) і збереженими security-headers на auth-сторінках обох локалей.

Ключова організаційна вимога: ввести спільну константу `NON_DEFAULT_LOCALES = ['en'] as const` як єдине джерело правди для (1) `routeRules` ключів `/en/auth/**`, (2) Task 3 server-middleware allow-list стрипання невалідних 2-літерних префіксів. Без цього parity-drift при додаванні 3-ї локалі гарантований (Decision 5).

Cookie config — explicit `cookieKey: 'i18n_lang'` (а не модульний default `i18n_redirected`) для прозорості у DevTools і збігу з smoke-checks user-spec'у. `cookieSecure` env-conditional (`process.env.NODE_ENV === 'production'`), `cookieMaxAge` НЕ передається — module default ~1y покриває user-spec вимогу 1 рік (verified Context7 — опція не існує у v9 `detectBrowserLanguage`, Decision 3 + User-Spec Deviations).

`routeRules` widening — без додавання `'/en/auth/**'` glob `'/auth/**'` не матчить `/en/auth/verify-email?token=...`, EN-юзер пройде через незахищений URL → token leak через crawlers/Referer (AC8, Decision 14). Reuse `NON_DEFAULT_LOCALES` константи через map/spread у `routeRules` гарантує parity при додаванні локалей.

Stub-локалі (мінімальний `{"common": {"placeholder": "x"}}`) потрібні щоб модуль стартував без помилок — повний контент заповнить Task 4 у Wave 3. Production-mode `silentTranslationWarn` + `silentFallbackWarn` (через `i18n.config.ts`) — defense-in-depth: build-gate ловить divergence, але runtime warning suppression уникає info disclosure через console (AC14, Decision 9).

## What to do

1. Встановити `@nuxtjs/i18n@^9` у `frontend/package.json` як dependency через `pnpm add` (НЕ devDep — модуль потрібен у runtime для SSR resolution).
2. Створити спільний файл `frontend/shared/i18n-locales.ts` з `export const NON_DEFAULT_LOCALES = ['en'] as const;` (щоб Task 3 middleware і `routeRules` мали єдине джерело правди).
3. Зареєструвати `'@nuxtjs/i18n'` у `nuxt.config.ts:modules` масиві.
4. Додати блок `i18n` до `nuxt.config.ts` з полями:
   - `locales` — масив з `uk` (file `uk.json`) і `en` (file `en.json`)
   - `defaultLocale: 'uk'`
   - `strategy: 'prefix_except_default'`
   - `langDir: 'locales'` (під Nuxt 4 з default `restructureDir: 'i18n'` resolve-иться у `frontend/i18n/locales/`)
   - `vueI18n: './i18n.config.ts'`
   - `detectBrowserLanguage` — об'єкт з ключами: `useCookie: true`, `cookieKey: 'i18n_lang'`, `cookieSecure` (env-conditional через `process.env.NODE_ENV === 'production'`), `cookieCrossOrigin: false`, `redirectOn: 'no prefix'`, `alwaysRedirect: false`, `fallbackLocale: 'uk'`. Опцію `cookieMaxAge` **не передавати** (модуль v9 її не приймає, Context7-verified).
5. Розширити `routeRules` у `nuxt.config.ts`: для кожного `code` з `NON_DEFAULT_LOCALES` додати ключ `'/{code}/auth/**'` з тими самими headers (`X-Robots-Tag: noindex, nofollow`, `Referrer-Policy: no-referrer`). Підхід — згенерувати dictionary через `Object.fromEntries` map по `NON_DEFAULT_LOCALES`, потім spread у `routeRules` поряд з existing `'/auth/**'`. Headers винести у named-константу (наприклад `AUTH_HEADERS`) щоб обидва ключі ділили один об'єкт. Існуючий ключ `'/auth/**'` зберегти.
6. Створити `frontend/i18n.config.ts`, що default-екпортує (через helper `defineI18nConfig`) об'єкт з полями: `legacy: false`, `fallbackLocale: 'uk'`, і двома production-conditional warning-suppressions (`silentTranslationWarn` + `silentFallbackWarn`) — обидва беруть значення `process.env.NODE_ENV === 'production'`.
7. Створити stub-файли `frontend/i18n/locales/uk.json` і `frontend/i18n/locales/en.json` з вмістом `{"common": {"placeholder": "x"}}`. Папка `i18n/` під Nuxt 4 restructureDir конвенцією (Risk: "Nuxt 4 path convention для langDir").
8. Запустити `pnpm install` у `frontend/`, перевірити що lockfile оновлений.
9. Виконати smoke-checks (див. Verification Steps).

### Smoke

Запустити dev-server у foreground або background: `cd frontend && pnpm dev` (port 3000). Дочекатися "Listening on http://localhost:3000". Виконати:

> **Agent-ergonomics note:** якщо запуск `pnpm dev` з агента non-feasible (TTY/lifecycle limits), fallback на `pnpm build` + `pnpm preview` (Nitro preview-server, той самий runtime), або зробити headless smoke через vitest з `setup` що піднімає Nitro server programmatically. Це не блокує задачу — будь-який з трьох варіантів покриває те саме observable behavior.

1. **Security headers parity на EN auth-сторінці:**
   ```bash
   curl -sI 'http://localhost:3000/en/auth/verify-email?token=x'
   ```
   Expect: response містить `X-Robots-Tag: noindex, nofollow` і `Referrer-Policy: no-referrer`. Перевіряє AC8 / Decision 14.

2. **Accept-Language → EN redirect + Set-Cookie:**
   ```bash
   curl -sI -H 'Accept-Language: en-US' 'http://localhost:3000/dashboard'
   ```
   Expect: `HTTP/1.1 302` з `Location: /en/...` і `Set-Cookie: i18n_lang=en` (cookie name explicit). Перевіряє Decision 3 + Decision 4.

3. **Невалідне cookie → fallback на default uk (no redirect):**
   ```bash
   curl -sI -H 'Cookie: i18n_lang=xx' 'http://localhost:3000/dashboard'
   ```
   Expect: НЕ редиректить (модуль ігнорує невалідну локаль), fallback на default `uk`. Перевіряє AC6 + Decision 4.

4. **Header parity на UK auth-сторінці (compare baseline для AC routeRules parity):**
   ```bash
   curl -sI 'http://localhost:3000/auth/login'
   ```
   Expect: ті самі `X-Robots-Tag: noindex, nofollow` і `Referrer-Policy: no-referrer` що й у запиті (1). Diff/grep двох responses не повинен показувати розбіжності у цих headers — це підтверджує routeRules parity без unit-тесту.

Зупинити dev-server після перевірки.

## Details

**Files:**
- `frontend/package.json` — додати `@nuxtjs/i18n@^9` у `dependencies` (через `pnpm add @nuxtjs/i18n@^9`, не вручну редагувати JSON). Поточний стан: 12 dependencies (nuxt, pinia, tanstack-vue-query тощо), 6 devDependencies. Конфлікт з Task 1 (Playwright) розрулений через Wave-1 solo (Decision 17).
- `frontend/nuxt.config.ts` — поточний стан: 27 рядків, `compatibilityDate`, `typescript.strict`, `modules: ['@pinia/nuxt', '@nuxtjs/tailwindcss']`, `css`, `runtimeConfig.public.apiBase`, `routeRules: { '/auth/**': { headers: { 'X-Robots-Tag': 'noindex, nofollow', 'Referrer-Policy': 'no-referrer' }}}`. Додати: imports `NON_DEFAULT_LOCALES`, `'@nuxtjs/i18n'` у modules, блок `i18n: {...}`, розширити `routeRules` через spread згенерованих ключів.
- `frontend/shared/i18n-locales.ts` (new) — single source of truth для не-default локалей. Експортує: `NON_DEFAULT_LOCALES = ['en'] as const` (literal-tuple) і derived type `NonDefaultLocale`. Імпортується і у `nuxt.config.ts` (для `routeRules` map), і пізніше у Task 3 server-middleware (allow-list).
- `frontend/i18n.config.ts` (new) — default-експорт через `defineI18nConfig` (auto-import від модуля). Поля: `legacy: false`, `fallbackLocale: 'uk'`, `silentTranslationWarn` і `silentFallbackWarn` — обидва env-conditional (`process.env.NODE_ENV === 'production'`).
- `frontend/i18n/locales/uk.json` (new stub) — `{"common": {"placeholder": "x"}}`. Папка `i18n/` створюється під Nuxt 4 restructureDir конвенцією. Заповнюється повністю у Task 4.
- `frontend/i18n/locales/en.json` (new stub) — `{"common": {"placeholder": "x"}}`. Те саме.

**Dependencies:**
- Залежить від Task 1 (Playwright bootstrap) — щоб уникнути merge-конфлікту на `frontend/package.json` (обидві задачі його модифікують, Decision 17).
- Жодних backend або infrastructure-залежностей.

**Edge cases:**
- Nuxt 4 restructureDir: модуль `@nuxtjs/i18n@^9` під Nuxt 4 за default-ом має `restructureDir: 'i18n'`. Це означає, що `langDir: 'locales'` resolve-иться **відносно** restructure-директорії (`frontend/i18n/`), фінальний шлях — `frontend/i18n/locales/` (а не `frontend/locales/`!). На відміну від Nuxt 3, де локалі лежали б у `frontend/locales/`. Mitigates Risk row "Nuxt 4 path convention для langDir" (tech-spec line 245).
- **Fallback якщо модуль не знаходить файли:** якщо `pnpm dev` фейлиться з помилкою про відсутні locale-файли (приклад: `Cannot find translation file en.json`), переключитися на абсолютний від srcDir шлях: `langDir: 'i18n/locales'` (без `restructureDir` resolve). Це робочий fallback — verify-step "langDir resolution verified" в AC покриває обидва варіанти.
- `cookieMaxAge` опція у v9 `detectBrowserLanguage` НЕ існує — спроба передати її проігнорується (Context7-verified, User-Spec Deviation approved 2026-05-08).
- `cookieSecure: true` на dev-сервері HTTP `localhost:3000` — браузер не збереже cookie. Тому env-conditional: `process.env.NODE_ENV === 'production'`.
- `routeRules` потребує glob-паттерн (не regex). Map з `NON_DEFAULT_LOCALES` має продукувати точно `'/en/auth/**'`, не `'/en/auth/*'` (одна `*` не матчить вкладені шляхи).
- Stub locale-content `{"common": {"placeholder": "x"}}` повинен мати ОДНАКОВИЙ key-set у обох файлах — інакше Task 4 prebuild-script упаде. Тому обидва файли ідентичні.
- На EN-redirect smoke-check Set-Cookie може мати `Secure` атрибут НЕ виставлений у dev (бо NODE_ENV=development). Це expected — перевіряємо тільки наявність `i18n_lang=en`.

**Implementation hints:**
- Не амбіційно імплементувати Task 3 (strip-invalid-locale) у цій задачі — Task 3 окремо, паралельно у Wave 2. Тут тільки модуль + config.
- `nuxt.config.ts` імпортує `NON_DEFAULT_LOCALES` із `shared/i18n-locales` (це вимога, не nice-to-have — для parity з Task 3).
- Для `routeRules`: побудувати dictionary не-default auth-ключів через `Object.fromEntries` map по `NON_DEFAULT_LOCALES`, потім spread у `routeRules` поряд з existing `'/auth/**'`. Виносити headers у named-константу (`AUTH_HEADERS`) щоб уникнути typo між двома записами — defense-in-depth для AC8.
- `vueI18n: './i18n.config.ts'` — шлях відносно frontend root. Без цього `legacy: false` (з `i18n.config.ts`) не застосується.
- `redirectOn: 'no prefix'` критично — без нього EN-cookie юзер з URL `/dashboard` не редиректнеться на `/en/dashboard` (UX непослідовний, Decision 4).
- Перевір що pnpm-lock.yaml попадає у git stage — без нього CI не зможе детерміністично відтворити білд.
- routeRules — Nuxt-internal механізм без публічного API, тому в задачі НЕ створюється unit-test для routeRules-структури. Покриття гарантується смоук-сценаріями (1) та (4) у Verification — порівняння header-output між `/auth/login` і `/en/auth/login`. Reviewer не повинен flag-ити відсутність unit-тесту для routeRules.

**Anchored to:** AC1 (модуль i18n + URL strategy + cookie + SameSite + Secure + 1y), AC8 (security headers parity на `/en/auth/**`), AC14 (parity-guard через спільну константу), Decisions 1, 2, 3, 4, 14, User-Spec Deviation про `cookieMaxAge`.
