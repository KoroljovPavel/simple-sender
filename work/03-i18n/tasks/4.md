# Task 4: Locale dictionaries + check-locales script

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](.claude/skills/code-writing/SKILL.md)

## Description

Заповнюємо locale-словники `frontend/i18n/locales/uk.json` і `en.json` повним набором ключів покриваючи **усі** видимі рядки інвентаризовані у code-research §1 (layouts, 5 auth-сторінок, dashboard, profile). Структура — top-level namespaces per Decision 8 (`auth`, `dashboard`, `profile`, `layout`, `validation`, `errors`, `common`, `brand`), camelCase англомовні ключі. UA-копія дослівно з existing `pages/`/`layouts/` файлів (без зміни сенсу); EN-копія — консервативно-формальним тоном. Окремий `errors` namespace будується під fallback chain `useApiError`-композебла (Decision 10): `errors.{contextKey}.{code}` → `errors.{contextKey}.generic` → `errors.generic`. Brand `Bot Funnel` зберігається ідентично у обох локалях під ключем `brand.name` (Decision 13).

Створюємо build-time валідатор `frontend/scripts/check-locales.mjs` per Decision 9 — Node ESM-скрипт, що парсить обидва JSON, рекурсивно збирає key-set через `Object.create(null)` (no prototype-pollution), фільтрує небезпечні ключі (`__proto__`, `constructor`, `prototype`), порівнює key-sets, exit(1) на: відсутній файл, невалідний JSON, розбіжність ключів. Підключається через npm-script `prebuild` у `frontend/package.json` — npm автоматично виконує `prebuild` перед `build`, тож CI ловить divergence до Nuxt-build.

Опційний smoke-test для самого скрипта — `frontend/scripts/check-locales.test.mjs` через `node:test` — покриває успіх parsing, missing file, invalid JSON, key-divergence, prototype-pollution payload `{"__proto__": {...}}` (асерт що runtime-prototype не змінюється). Цей набір тестів — основа TDD для скрипта: **пишемо тести FIRST з prototype-pollution фікстурою, потім реалізуємо скрипт**.

Anchored to AC2 (однаковий набір ключів між локалями), AC19 (build падає на missing/corrupted JSON), Decisions 8/9/13. Risks: missing key між uk і en (build-gate ловить), prototype pollution через `__proto__` payload у JSON (Object.create(null) + filter).

## What to do

1. Сформувати повний key-set per Decision 8 namespaces із code-research §1.1-§1.4 (всі strings → keys). Включити:
   - `brand.name` = `"Bot Funnel"` в обох локалях (constant per Decision 13).
   - `layout.default.*` (Logout, Dashboard, Profile + brand wrap).
   - `auth.login.*`, `auth.register.*`, `auth.forgotPassword.*`, `auth.resetPassword.*`, `auth.verifyEmail.*` — title, labels, placeholders, button states (`submit`, `submitting`), helper texts, links.
   - `auth.verifyEmail.cooldown` з `{seconds}` placeholder для interpolation.
   - `dashboard.*` (banner, link, welcome with `{name}` placeholder for conditional, description).
   - `profile.*` — 5 секцій (personalData, emailVerification, changePassword, sessions, deleteAccount) + modal labels + interpolated cooldowns (`{seconds}`).
   - `validation.*` — Zod messages (emailFormat, passwordMin8, passwordHasLetter, passwordHasDigit, passwordsDoNotMatch, nameRequired, nameTooLong, currentPasswordRequired).
   - `errors.{contextKey}.{code}` per code-research §2 mapping table — конкретні коди (401, 403, 409, 429) для кожного context (`login`, `register`, `forgotPassword`, `resetPassword`, `verifyEmail`, `changePassword`, `terminateSessions`, `deleteAccount`, `saveName`).
   - `errors.{contextKey}.generic` для кожного context.
   - `errors.generic` — final fallback.
   - `common.*` — wide reuse: `cancel`, `save`, `edit`, `email`, `password`, тощо.

2. Заповнити `frontend/i18n/locales/uk.json` — UA-копія з existing files (дослівно). Перевизначити hardcoded English `Logout` / `Dashboard` / `Profile` у `default.vue` на UA (`Вийти` / `Дашборд` / `Профіль`) — це content change, явно зафіксований у code-research §10 п.7.

3. Заповнити `frontend/i18n/locales/en.json` — однаковий набір ключів, EN-переклади консервативно-формальним тоном (e.g. `Sign in`, `Sign up`, `Logout`, `Forgot password?`, `Send reset link`, `Invalid email format`, `Too many attempts. Try again in 15 minutes`).

4. **Спочатку** написати `frontend/scripts/check-locales.test.mjs` з тестами per Testing Strategy (запустити, переконатись що падають). Кейси:
   - parse OK — обидва файли з однаковими ключами → exit 0.
   - missing file → exit 1.
   - invalid JSON → exit 1.
   - key-divergence (uk has key X, en doesn't) → exit 1, повідомлення з missing key.
   - prototype-pollution payload `{"__proto__": {"polluted": "yes"}}` → safe (`({}).polluted === undefined` після виконання скрипта).

5. Реалізувати `frontend/scripts/check-locales.mjs`:
   - Resolve paths відносно скрипта (НЕ CWD), щоб npm-prebuild працював незалежно від директорії запуску.
   - Парсити обидва файли через `fs.readFileSync` + `JSON.parse` всередині try/catch (catch → console.error + `process.exit(1)`).
   - Рекурсивно зібрати key-set: створити збиральник через `Object.create(null)`, при traverse фільтрувати небезпечні ключі (`__proto__`, `constructor`, `prototype`) — НЕ робити recursive descent у їхні value.
   - Порівняти key-sets: різниця → console.error з переліком missing keys + `process.exit(1)`.
   - Exit 0 при success.

6. Додати npm-script `prebuild` у `frontend/package.json`: `"prebuild": "node scripts/check-locales.mjs"`. Зберегти існуючі scripts без модифікації.

7. Запустити skip-test → reality check:
   - `cd frontend && node scripts/check-locales.mjs` → exit 0.
   - тимчасово видалити будь-який ключ з `en.json` → script exits 1 з missing-keys повідомленням → відновити.
   - `mv en.json en.json.bak && node scripts/check-locales.mjs; echo $?` → 1 → відновити.
   - `pnpm build` проходить (prebuild + Nuxt build).

## Details

**Files:**
- `frontend/i18n/locales/uk.json` — поточний стан: stub `{"common": {"placeholder": "x"}}` (створений у Task 2). Замінити повним nested JSON-словником з усіма namespaces per Decision 8. UA-копія дослівно з existing pages/layouts.
- `frontend/i18n/locales/en.json` — поточний стан: stub. Замінити повним JSON з ідентичною key-структурою + EN-переклади консервативно-формальним тоном.
- `frontend/scripts/check-locales.mjs` — новий файл. ESM Node-script (~40-60 LOC) per Decision 9.
- `frontend/scripts/check-locales.test.mjs` — новий, опційний (recommended). Через `node:test` (built-in, без додаткових deps).
- `frontend/package.json` — додати ключ `"prebuild": "node scripts/check-locales.mjs"` у `scripts` блок (поточний стан: `dev`, `build`, `generate`, `preview`, `test`). НЕ додавати нові dependencies — `node:test` built-in від Node 24 (`.nvmrc` пінує).

**Dependencies:**
- Залежить від Task 2 — Task 2 створює директорію `frontend/i18n/locales/` зі stub-файлами + базовий i18n module config. Task 4 наповнює stubs.
- Не блокує Wave 4 tasks (8, 9, 10) які споживають ці словники через `t()` — Wave 4 чекає Wave 3 повністю.

**Edge cases:**
- **Prototype pollution через JSON-payload.** JSON-парсер дозволяє `__proto__` як ключ; наївний `for-in` обхід може через `Object.assign` чи аналог заразити prototype chain. Mitigation: `Object.create(null)` для key-set + filter небезпечних ключів перед recursion. Test fixture явно перевіряє `({}).polluted === undefined` після script-run.
- **Path resolution.** Якщо `node scripts/check-locales.mjs` запускається з різних CWD (npm prebuild vs. ручний запуск), relative paths ламаються. Використовувати `new URL('../i18n/locales/uk.json', import.meta.url)` (anchor до файлу скрипта).
- **JSON-parse помилки.** Catch SyntaxError, логувати з file path і повідомленням, exit 1.
- **Both files identical empty `{}`.** Технічно не divergent, але білд впаде на runtime missing-key. Не покривається цим скриптом — це окремий gate (runtime fallback chain через `te()` per Decision 10). Не додавати "min keys count" check — overkill для feature scope.
- **Brand `Bot Funnel`.** Однакове значення в обох локалях. Скрипт не верифікує **values** — тільки key-sets. Це OK per AC scope.
- **Interpolated keys** (`{seconds}`, `{name}`). Vue I18n синтаксис — це лише strings з `{...}` placeholders, не підлягають перевірці у script (vue-i18n валідує на runtime).

**Implementation hints:**
- `node:test` runner у Node 24 — імпортується з `node:test`, асерції з `node:assert/strict`. Запуск: `node --test scripts/check-locales.test.mjs`.
- Для test fixture з prototype-pollution payload — створити tmp-директорію з тимчасовими `uk.json`/`en.json`, передати script кастомним arg (або через env-var `LOCALES_DIR`), перевірити exit code через `child_process.spawnSync`.
- Альтернативно — експортувати pure-function `collectKeySet(obj)` з самого `check-locales.mjs` і тестувати її напряму без spawn (швидше, простіше). Скрипт стає як module + main: при прямому запуску (`import.meta.url === ...`) виконує main, інакше — тільки експортує функції.
- key-set storage: `const keys = Object.create(null);` + `keys[path] = true;` (path виду `auth.login.title`).
- Recursion: при traverse якщо `key === '__proto__' || key === 'constructor' || key === 'prototype'` — `continue` (skip). Це додатковий захист поверх `Object.create(null)`.
- Коли різниця знайдена — друкувати обидві сторони ({uk-only, en-only}) для зручності debugging.
- Перевірити що скрипт працює з `prefix_except_default` локалями — нормально для будь-якої кількості (eager-load обох локалей per Decision 7 не впливає на скрипт).
- EN-копія: дотримуватись формального тону. Приклади: `Sign in` (не `Login`), `Forgot password?` (не `Forgot your password?`), `Invalid email format` (не `Please enter a valid email`). Це не requirement для CI, але AC18 (test brittleness) захищає тільки UA-asserts — EN не тестується автоматично, тому якість залежить від ручного review.
