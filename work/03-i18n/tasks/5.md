# Task 5: useApiError factory composable

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Створити `frontend/composables/useApiError.ts` per Decision 10 — factory composable, який централізує мапінг HTTP-error → локалізоване повідомлення. На сьогодні логіка `e?.statusCode ?? e?.status ?? e?.response?.status` дублюється 8 разів у `pages/auth/*.vue` і `pages/profile.vue` з hardcoded UA-рядками; цей composable замінює всі дублікати на спільний handler з `te()`/`t()` lookup-ом.

Ключові вимоги:
- Factory pattern: `useApiError()` викликається у `<script setup>` (правильний lifecycle для `useI18n()`); повертає handler `(error: unknown, contextKey: string) => string` який вже можна викликати у catch-блоках.
- Status extraction: `error?.statusCode ?? error?.status ?? error?.response?.status`. Якщо результат не number — трактувати як "no status" і fallback одразу на `errors.generic`.
- Lookup chain через `te()` (translation-exists check) ПЕРЕД `t()`: `errors.{contextKey}.{status}` → `errors.{contextKey}.generic` → `errors.generic`. Без `te()`-перевірки vue-i18n у production-mode мовчки повертає key-string (`silentTranslationWarn: true` per Decision 9), що порушить UX.

Vitest spec покриває fallback gradient + extraction з різних error-форм; mock `useI18n` через `mockNuxtImport` за патерном з `tests/middleware/auth.global.spec.ts`.

## What to do

   - Повертає функцію `(error: unknown, contextKey: string) => string`.
   - Handler виконує status-extraction з error-об'єкта і `te()`/`t()` lookup chain.
2. Створити `frontend/tests/composables/useApiError.spec.ts` з повним покриттям fallback-gradient і status-extraction (TDD — пишемо тести ДО імплементації).

## Details

**Files:**
- `frontend/composables/useApiError.ts` (new) — factory composable. Naming і експорт за патерном `frontend/composables/useApi.ts` (named export, no default). Імпорт `useI18n` — auto-imported Nuxt-композіт, не пишемо explicit `import`.
- `frontend/tests/composables/useApiError.spec.ts` (new) — Vitest spec. Розташування — нова піддиректорія `tests/composables/` (зараз існують `tests/middleware`, `tests/pages`, `tests/stores` — додаємо нову, відповідно структурі).

**Dependencies:**
- Task 2 (i18n module bootstrap) — має бути завершений; `useI18n` має бути auto-imported через `@nuxtjs/i18n` plugin.
- `errors.*` namespace у `frontend/i18n/locales/{uk,en}.json` буде заповнений у Task 4 (паралельно у тій же хвилі) — для тестів цього таска це не блокер, бо `t`/`te` мокаються.

**Edge cases:**
- `error === null` / `error === undefined` → `error?.statusCode` гасить chain, status не number → fallback на `errors.{contextKey}.generic` (через `te()`) → `errors.generic`.
- `error.status === 0` (мережева помилка fetch) — це валідний number, lookup `errors.{contextKey}.0` через `te()` поверне false (ключа немає) → fallback на context-generic. Не треба окремої логіки.
- `error.statusCode === '401'` (string) — `??` пропустить, але числовий ключ `errors.login.401` не співпаде з string interpolation `errors.login.${statusCode}` без приведення. Decision: приводити до number через `Number()` і перевіряти `Number.isFinite(...)` перед lookup-ом; інакше status трактується як "no status".
- `contextKey` порожній рядок або undefined — теоретично не має статись, бо контракт `string`, але defensively → fallback на `errors.generic`.
- Production-mode silent missing-key — Decision 9 виставляє `silentTranslationWarn: true`, тому `t()` без `te()`-prefix мовчки поверне key-string. Hence `te()`-gating є обов'язковим — це не оптимізація.

**Implementation hints:**
- Структура factory:
  - `useApiError()` викликає `const { t, te } = useI18n()` на верхньому рівні factory (НЕ всередині повернутого handler — інакше lifecycle проблема не вирішена).
  - Повертає arrow або named function; внутрішня логіка читає `error?.statusCode ?? error?.status ?? error?.response?.status`, нормалізує через `Number()`/`Number.isFinite`, будує candidate keys array (`[errors.${ctx}.${status}, errors.${ctx}.generic, errors.generic]` з пропуском першого якщо status невалідний), пробігає `te()` → перший true → `t()`.
- Mock pattern для тестів — за `tests/middleware/auth.global.spec.ts:1-15`:
  ```
  const { tMock, teMock } = vi.hoisted(() => ({ tMock: vi.fn(), teMock: vi.fn() }))
  mockNuxtImport('useI18n', () => () => ({ t: tMock, te: teMock }))
  ```
  У кожному тесті `teMock.mockImplementation((key) => key === 'errors.login.401')` тощо; асертимо `tMock` був викликаний з очікуваним ключем і НЕ викликаний з ключами де `te` повернув false.
- TypeScript: `error: unknown` — обов'язково розпаковувати через type-narrowing або optional-chaining з `as` (`(error as { statusCode?: unknown })?.statusCode`). Avoid `any`.
- Не імпортувати `useI18n` з `vue-i18n` напряму — Nuxt auto-import з `@nuxtjs/i18n` дає правильний bound version.
