# Task 6: LangSwitcher component + Vitest

## Required Skills

Перед виконанням задачі завантаж:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](.claude/skills/code-writing/SKILL.md)

## Description

Створюємо `frontend/components/LangSwitcher.vue` — текстовий toggle `UK | EN` (НЕ dropdown), який юзер натискає, щоб перемкнути локаль фронтенду. Per Decision 12 — click-handler викликає `await setLocale(code)` (canonical v9 pattern: модуль atomically оновлює cookie `i18n_lang`, locale state, `<html lang>` через `useLocaleHead`, і робить auto-navigate на локалізований path під strategy `prefix_except_default`). Активна локаль виділяється Tailwind-класами для очевидного візуального стану.

Паралельно пишемо `frontend/tests/components/LangSwitcher.spec.ts` (Vitest + `@nuxt/test-utils/runtime`) — асерції на observable behavior: `setLocale('en')` був викликаний з правильним аргументом, `useCookie('i18n_lang').value === 'en'` після click (per AC15 "cookie оновлений" — асерція явно вимагається user-spec'ом), активна локаль виділена. `useI18n().setLocale` мокаємо як spy через `mockNuxtImport`, `useCookie` теж мокаємо — асертимо value, не deep mock-плумбінг. TDD-flow: пишемо спек FIRST, переконуємось що падає, потім імплементуємо компонент.

Компонент потім буде вмонтовано у Task 8 (default.vue топбар + auth.vue absolute top-right) — у цьому таску ми створюємо лише сам файл компонента і spec. Якірні AC — AC9 (toggle UK | EN, клік перемикає атомарно: cookie + URL + `<html lang>` + контент) і AC15 (cookie оновлений). Risk anchor — race-condition між cookie write і navigation, mitigated через `await setLocale(code)` (atomic у модулі — Decision 12).

## What to do

1. Написати Vitest spec `frontend/tests/components/LangSwitcher.spec.ts` ДО реалізації (TDD):
   - Mock `useI18n` через `mockNuxtImport` — повертає `{ locale: ref('uk'), locales: [{code:'uk',...},{code:'en',...}], setLocale: vi.fn() }`. Spy на `setLocale` доступний через `vi.hoisted`.
   - Mock `useCookie` через `mockNuxtImport` — повертає `ref(<value>)` (mutable), щоб після `setLocale('en')` тест міг асертити `cookieRef.value === 'en'`. Mock-implementation сам ставить ref-value у `setLocale` spy: `setLocale.mockImplementation(async (code) => { localeRef.value = code; cookieRef.value = code })`.
   - Кейси (Acceptance Criteria + Testing Strategy):
     - render: компонент показує і `UK` і `EN` як clickable elements.
     - active highlight: при `locale.value === 'uk'` — UK має активний Tailwind-клас (наприклад `font-semibold` або `text-primary`), EN — звичайний; перевіряємо via class-наявність на конкретному елементі (data-test-attribute рекомендовано: `data-test="lang-uk"` / `data-test="lang-en"`).
     - click EN → `setLocale` викликаний рівно один раз із аргументом `'en'`.
     - click EN + await flush → cookie ref value стає `'en'` (per AC15).
     - click на currently-active locale (UK при locale='uk') → no-op або idempotent: `setLocale` НЕ викликаний, або якщо викликаний — без помилки і без додаткових side-effects.
2. Запустити спек → переконатись що всі кейси падають (`cd frontend && pnpm test tests/components/LangSwitcher.spec.ts`).
3. Створити `frontend/components/LangSwitcher.vue`:
   - `<script setup lang="ts">`: `const { locale, locales, setLocale } = useI18n()`. Хелпер `isActive(code)` повертає `code === locale.value`. Click-handler `async function switchTo(code) { if (code === locale.value) return; await setLocale(code) }`.
   - Template: дві текстові кнопки `UK` і `EN` (рекомендовано `<button type="button">` для accessibility — keyboard focusable), розділені `|`. Активна — Tailwind-клас (`font-semibold text-primary` або equivalent), неактивна — м'якший колір. `data-test="lang-{code}"` для тестів. `aria-label` per кнопка для screen reader, `aria-current` для активної.
   - НЕ використовувати dropdown / select. НЕ використовувати explicit `navigateTo(switchLocalePath(code))` — тільки `setLocale(code)` (Decision 12: auto-navigate canonical pattern, separating дублює внутрішню роботу модуля і додає race-condition surface).
4. Запустити спек повторно → всі кейси проходять.
5. Перевірити `pnpm test` — без regressions у інших specs.

## TDD Anchor

Тести, які потрібно написати ДО реалізації. Пишемо → запускаємо → переконуємось що падають → пишемо компонент → переконуємось що проходять.

- `tests/components/LangSwitcher.spec.ts::renders both UK and EN` — компонент рендерить обидва toggle-options як clickable elements (`UK` + `EN` видимі у DOM).
- `tests/components/LangSwitcher.spec.ts::active locale has highlight class` — при `locale.value === 'uk'` активна кнопка UK має очікуваний Tailwind-клас (наприклад `font-semibold` чи `text-primary`), а EN — ні; перевіряти через `wrapper.find('[data-test="lang-uk"]').classes()` matcher.
- `tests/components/LangSwitcher.spec.ts::click EN calls setLocale with en` — `wrapper.find('[data-test="lang-en"]').trigger('click')` → `setLocale` викликаний рівно `1` раз з аргументом `'en'` (`expect(setLocaleSpy).toHaveBeenCalledExactlyOnceWith('en')`).
- `tests/components/LangSwitcher.spec.ts::click EN updates cookie i18n_lang to en` — після click + `await settle()` `useCookie('i18n_lang').value === 'en'` (per AC15). Mock-implementation `setLocale` оновлює cookie ref як side-effect.
- `tests/components/LangSwitcher.spec.ts::click on currently-active locale is idempotent` — `locale='uk'`, click UK → `setLocale` НЕ викликаний (early-return guard), компонент не throw.

Структура: дві `<button>` з `data-test="lang-{code}"`, click → `switchTo(code)`. Tailwind-класи для active state. **Поточний стан:** директорія `frontend/components/` не існує — створити при імплементації.
- `frontend/tests/components/LangSwitcher.spec.ts` (NEW) — Vitest spec. Імпорти: `describe, it, expect, beforeEach, vi` з `vitest`; `ref` з `vue`; `mockNuxtImport, mountSuspended` з `@nuxt/test-utils/runtime`; `settle` з `../../helpers/settle`. **Поточний стан:** директорія `frontend/tests/components/` не існує — створити при імплементації.

**Dependencies:**
- Task 2 (i18n bootstrap) має бути done — `@nuxtjs/i18n@^9` встановлений, `useI18n` auto-import працює, locale stub-файли існують.
- Жодних додаткових npm-пакетів не треба (Vitest + `@nuxt/test-utils` + `@vue/test-utils` + `happy-dom` вже у devDependencies).

**Edge cases:**
- Click на active locale (UK при locale='uk') — early-return у handler, `setLocale` НЕ викликаний (захист від redundant network/state операцій + AC покриття).
- `setLocale` async — handler має `await` (per Decision 12 canonical pattern); тест використовує `await wrapper.find(...).trigger('click')` + `await settle()` для flush.
- Mock cookie ref має бути mutable (`ref(...)` з `vue`), щоб mock-implementation `setLocale` міг присвоїти `cookieRef.value = code` і тест побачив update.
- `locales` array у mock — об'єкти `{code: 'uk', ...}` / `{code: 'en', ...}`, не просто строки (per `@nuxtjs/i18n` v9 typings — модуль augment'ить контракт).
- Tailwind-класи для active state — використовуй existing project patterns (див. ux-guidelines.md). Рекомендовано: `font-semibold text-primary` для активної, `text-muted-foreground hover:text-foreground` для неактивної.
- Race-condition mitigation: `await setLocale(code)` — модуль atomically оновлює cookie + locale state + URL. Не розривати на ручні `navigateTo` + `switchLocalePath` (Decision 12 alternative rejected).

**Implementation hints:**
- `vi.hoisted({ setLocaleSpy: vi.fn(), localeRef: ref('uk'), cookieRef: ref('uk') })` — pattern з existing tests (login.spec.ts ll. 6-10).
- `mockNuxtImport('useI18n', () => () => ({ locale: localeRef, locales: [{code:'uk', name:'Українська'}, {code:'en', name:'English'}], setLocale: setLocaleSpy }))`.
- `mockNuxtImport('useCookie', () => (key: string) => key === 'i18n_lang' ? cookieRef : ref(null))`.
- `setLocaleSpy.mockImplementation(async (code) => { localeRef.value = code; cookieRef.value = code })` — у `beforeEach`, щоб side-effect моделював реальну module-поведінку.
- Не імпортувати `useI18n` з `vue-i18n` напряму — `@nuxtjs/i18n` робить auto-import та augment'ить контракт (locales array, setLocale signature).
- Component `<button>` з `type="button"` — щоб не subm'ититись як form у разі вмонтування у форму (defensive).
- `aria-label` per кнопка: `aria-label="Switch language to Ukrainian"` / `... to English` — для screen reader.
- Для активної кнопки додатково `aria-current="true"` — semantic indicator активного стану.
- `setLocaleSpy.mockReset()` у `beforeEach` (а потім re-set implementation) — щоб counter і calls очищувались між тестами.
