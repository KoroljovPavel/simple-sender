# Task 7: Auth middleware locale-aware refactor

## Required Skills

Перед выполнением задачи загрузи:
- `/skill:code-writing` — [skills/code-writing/SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Поточний `frontend/middleware/auth.global.ts` визначає публічний auth-маршрут через `to.path.startsWith('/auth/')`. Під strategy `prefix_except_default` цей префіксний матч ламається на en-локалі: `/en/auth/login` не починається з `/auth/`, тож:
- авторизований юзер на `/en/auth/login` НЕ редиректиться на `/dashboard` (втрата safety net);
- неавторизований юзер на `/en/dashboard` редиректиться на `/auth/login` без префікса, втрачаючи локаль.

Декомпонуємо middleware на іменну перевірку через композабл `useRouteBaseName()` (Decision 6, AC10). Композабл повертає функцію, яка дає базове ім'я роуту незалежно від локалі (`auth-login`, `dashboard`, `auth-register` тощо — модуль `@nuxtjs/i18n` генерує internal `auth-login___uk` / `auth-login___en`, а композабл стрипає `___locale`). Whitelist публічних роутів задаємо як explicit array `PUBLIC_ROUTE_NAMES`. Якщо `getBaseName(to)` повертає `undefined` (route не зарезолвлено — catch-all 404 чи невідомий роут) — поводимось fail-closed: трактуємо як protected, бо security-неприйнятно пропускати неідентифіковані шляхи. Усі `navigateTo()` обгортаємо у `useLocalePath()`, щоб редирект уважав поточну локаль (uk → `/auth/login` і `/dashboard`; en → `/en/auth/login` і `/en/dashboard`).

Тести у `frontend/tests/middleware/auth.global.spec.ts` переписуємо: path-based фікстури замінюємо на нові кейси, які мокають `useRouteBaseName` і `useLocalePath` через `mockNuxtImport`. Це авторизований виняток з AC18 (Decision 16): існуючі test-файли не модифікуємо, але `auth.global.spec.ts` — explicit exception, бо логіка middleware принципово змінюється.

## What to do

1. Замінити логіку у `frontend/middleware/auth.global.ts`:
   - Усередині `defineNuxtRouteMiddleware`-handler викликати `useRouteBaseName()` (composable повертає функцію, по конвенції `getRouteBaseName` — локально alias-уй як `getBaseName` для краткості).
   - Оголосити `const PUBLIC_ROUTE_NAMES = ['auth-login', 'auth-register', 'auth-forgot-password', 'auth-reset-password', 'auth-verify-email'] as const` на module-рівні (поза handler-ом).
   - Для кожного `to` обчислити `baseName = getBaseName(to)`.
   - Якщо `baseName` undefined → fail-closed (трактувати як protected: якщо неавторизований — редирект на login).
   - Якщо `baseName` входить у `PUBLIC_ROUTE_NAMES` → це public auth route.
   - Зберегти існуючу гідрацію `authStore.fetchUser()` з try/catch (transient errors не падають middleware).
   - Усі `navigateTo(...)` виклики обгорнути у `useLocalePath()` — composable повертає функцію, тому збережи у локальну змінну `const localePath = useLocalePath()` і викликай `navigateTo(localePath('/dashboard'))` / `navigateTo(localePath('/auth/login'))`.
2. Переписати `frontend/tests/middleware/auth.global.spec.ts` під нову логіку:
   - Замість path-based makeRoute — генерувати route-об'єкт, який проходить у `useRouteBaseName`-mock; mock повертає `getBaseName` що для конкретного `to` віддає потрібний baseName.
   - Через `mockNuxtImport` замокати `useRouteBaseName` (повертає `() => baseName`) і `useLocalePath` (повертає `(path) => prefixedPath`).
   - Зберегти існуючий mock pattern для `navigateTo` і `useApi` (`vi.hoisted`).
   - Покрити кейси з TDD Anchor (нижче).
3. Запустити test-suite — переконатися що всі нові кейси проходять, інші existing specs (auth pages тощо) не зачеплені.

- `tests/middleware/auth.global.spec.ts::unauthenticatedWithValidSession_publicAuthRouteByName_hydratesAndRedirectsToDashboard` — `apiMock.mockResolvedValueOnce(ACTIVE_USER)`, `getBaseName` → `auth-login` → middleware гідрує користувача і редиректить на dashboard.

## Details

**Files to modify:**

- `frontend/middleware/auth.global.ts` — поточний код використовує `function isPublicAuthRoute(path)` + `path.startsWith('/auth/')`. Замінити на `useRouteBaseName()` композабл-виклик усередині handler-а, додати `PUBLIC_ROUTE_NAMES` constant на module-рівні, обернути обидва `navigateTo` у `useLocalePath()`. Зберегти `try/catch` навколо `authStore.fetchUser()` — Pinia store API без змін, його не чіпаємо.
- `frontend/tests/middleware/auth.global.spec.ts` — поточні 6 кейсів написані під path-логіку (`/dashboard`, `/auth/login`). Переписати під name-based: додати `mockNuxtImport('useRouteBaseName', ...)` і `mockNuxtImport('useLocalePath', ...)`; адаптувати `makeRoute` під нові потреби (можна спростити — для name-based логіки `to.path` стає менш важливим, бо рішення приймається через mock-метод).

**Dependencies:**

- Task 2 (i18n module bootstrap) — встановлений `@nuxtjs/i18n@^9`, налаштована strategy `prefix_except_default` і locale config; без модуля композабли `useRouteBaseName` / `useLocalePath` недоступні. Тому wave 3 (після Wave 2).
- Не чіпає `frontend/stores/auth.ts`, `frontend/composables/useApi.ts`, `frontend/types/user.ts` — їх API і типи стабільні (per code-research §7).

**Edge cases:**

- Catch-all / 404 / невідомий роут: `getBaseName(to)` повертає `undefined`. Fail-closed: трактуємо як protected (unauthed → login).
- Гідрація через cookie session: `authStore.user === null` спочатку, але session valid — `await authStore.fetchUser()` мутує state, далі логіка йде нормально (як зараз).
- Transient backend error (500) при `fetchUser`: catch ковтає, `authStore.user` залишається null → unauthed → редирект на login (захист).
- Локаль uk (default) під `prefix_except_default`: `useLocalePath('/dashboard')` повертає `/dashboard` без префікса; локаль en — `/en/dashboard`. Mock у тестах має це симулювати окремими test-runs з різними mock implementations.
- SSR vs client: middleware виконується і на сервері, і на клієнті. Композабли `useRouteBaseName`/`useLocalePath` працюють у обох контекстах через `@nuxtjs/i18n` (документовано у v9).

**Implementation hints:**

- Структура middleware: `const getBaseName = useRouteBaseName()` і `const localePath = useLocalePath()` викликаються один раз на старті handler-а — тримай їх як локальні константи, не викликай повторно.
- `PUBLIC_ROUTE_NAMES` оголошуй на module-рівні (поза `defineNuxtRouteMiddleware`), щоб не пересоздавати на кожен виклик.
- Перевірка whitelist: `(PUBLIC_ROUTE_NAMES as readonly string[]).includes(baseName)` — `as readonly string[]` потрібен щоб TS не звужував до literal-types при `.includes()`.
- Mock-shape для тестів: `mockNuxtImport('useRouteBaseName', () => () => mockGetBaseName)` де `mockGetBaseName: vi.Mock<[unknown], string | undefined>`. Аналогічно `mockNuxtImport('useLocalePath', () => () => mockLocalePath)`.
- Збережи pattern `vi.hoisted` для `navigateToMock` і `apiMock` (як у поточному файлі) — він взаємодіє з `mockNuxtImport`.
- Поточний spec використовує `useState<unknown>('auth-user').value = null/ACTIVE_USER` для прямої мутації стейту Pinia — лиши цей механізм, він не залежить від локальної логіки.
- TS-типи: middleware приймає `to: RouteLocationNormalized` з `vue-router` — імпорт залишається.
- Decision 6 явно говорить "fail-open на undefined — security-неприйнятно". Тому fail-closed гілка обов'язкова і має бути explicit-test'нута.
