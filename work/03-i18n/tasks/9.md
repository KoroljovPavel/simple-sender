# Task 9: Auth pages translation (5 files)

## Required Skills

Before starting, load:
- `/skill:code-writing` — [SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Translate the five auth pages — `pages/auth/login.vue`, `register.vue`, `forgot-password.vue`, `reset-password.vue`, `verify-email.vue` — from hardcoded UA literals to `t()` calls backed by the locale dictionaries built in Task 4. Replace all hand-rolled HTTP-error-to-string mappings in `catch` blocks with the `useApiError()` factory composable created in Task 5. Switch every `<NuxtLink>` to `<NuxtLinkLocale>` and wrap every `navigateTo()` target with `localePath()` so links and redirects respect the active locale prefix under strategy `prefix_except_default`.

A subtle but important constraint (Decision 11): each Zod schema in these pages is constructed at `<script setup>` time and "bakes in" whatever string `t()` returned at that moment. Without restructuring, switching the locale live (via `<LangSwitcher>` from Task 6) leaves stale UA validation messages in the closure. To avoid that, every Zod schema must be wrapped in `computed(() => z.object({...}))` and fed through `toTypedSchema(unref(...))` — schema rebuilds when `locale.value` changes, validation messages re-render in the new language.

The `verify-email.vue` page additionally uses interpolated cooldown text (`Зачекайте ${cooldownSecondsLeft} с` and `Спробуйте знову через {{ cooldownSecondsLeft }} с`). These become `t('auth.verifyEmail.cooldown', { seconds: cooldownSecondsLeft })` with proper named-placeholder interpolation in the locale JSON.

This task closes AC11 (auth pages localized), AC12 (Zod messages localized + locale-reactive), AC13 (API errors via shared composable), and contributes to the `<NuxtLink>` → `<NuxtLinkLocale>` and `navigateTo`-wrap acceptance criteria. Existing Vitest specs for these pages were authored against UA defaults and continue to pass unchanged (per AC18 + Decision 16) — the implementation must keep UA literals identical to what the existing tests assert.

**Dependencies:**
- Task 4 (locale dictionaries `uk.json` + `en.json`) — must be complete first; this task assumes the auth namespace is fully populated.
- Task 5 (`useApiError()` composable) — must be complete first; this task assumes `errors.login.*`, `errors.register.*`, `errors.resetPassword.*` namespaces exist with `429`/`401`/`403`/`409`/`400`/`generic` keys per the API-error mapping in code-research §2.
- Task 6 (`<LangSwitcher>` component) — not a hard dependency, but the manual user-verify needs the switcher to live-toggle locale on these pages.

**Edge cases:**
- Zod `.refine` callbacks reference outer fields (e.g. `data.password === data.confirmPassword`) — the closure captures `data` at validation time, but the message is read from `t()` at schema-construction time; `computed` wrapping handles this correctly because the entire `z.object({...}).refine(...)` chain is re-built when `locale.value` changes.
- vee-validate's `useForm({ validationSchema: toTypedSchema(unref(schemaComputed)) })` accepts a static schema, not a reactive one. To make validation re-run on locale change, the second argument to `toTypedSchema` is the *unwrapped current* schema; vee-validate's `validationSchema` accepts a Ref/computed in v4 — verify by reading vee-validate v4 + @vee-validate/zod docs (Context7 query if uncertain). If `validationSchema` does not accept reactive input directly, pass `computed(() => toTypedSchema(schemaComputed.value))` or invoke `setSchema()` from `useForm` on locale change.
- `verify-email.vue` cooldown `setInterval` runs every second — make sure interpolated `t('...cooldown', { seconds })` is re-evaluated each tick (it is, since the template re-renders when `cooldownSecondsLeft` ref updates).
- The `Увійти →` arrow on success CTA: include the arrow inside the locale value (e.g. `"signIn": "Sign in →"`) for consistency, OR keep the arrow as a separate `<span>` outside `t()`. Prefer the former — simpler markup, matches existing UA literal.
- `forgot-password.vue` silent catch: do not introduce visible error UI even with `useApiError()` at hand — that would leak information about whether the email exists. Keep `submitted.value = true` regardless of catch outcome.
- `<NuxtLinkLocale>` accepts the same `to` prop shape as `<NuxtLink>`; class/href bindings unchanged.
- `localePath()` on object-form `navigateTo`: only the `path` field is wrapped; `query`, `hash`, etc. remain untouched.

**Implementation hints:**
- vee-validate v4 + `@vee-validate/zod` reactive schema pattern (per code-research §10):
  ```ts
  const schema = computed(() => z.object({
    email: z.string().email(t('validation.emailFormat')),
    password: z.string().min(8, t('validation.passwordMin')),
    rememberMe: z.boolean(),
  }))
  const { defineField, handleSubmit, ... } = useForm({
    validationSchema: computed(() => toTypedSchema(schema.value)),
    initialValues: { ... },
  })
  ```
- `useApiError()` factory call site:
  ```ts
  const apiError = useApiError()
  // ... inside catch:
  submitError.value = apiError(e, 'login')
  ```
- For the cooldown button in verify-email.vue, the `t()` call sits inside a Vue template expression (`{{ cooldownSecondsLeft > 0 ? t('auth.verifyEmail.cooldown', { seconds: cooldownSecondsLeft }) : t('auth.verifyEmail.resend') }}`).
- Auto-imports under @nuxtjs/i18n v9: `useI18n`, `useLocalePath`, `<NuxtLinkLocale>` are auto-registered. Explicit `const { t } = useI18n()` is still recommended for IDE intellisense and for clarity in code review.
- When locale JSON keys are uncertain, cross-reference Task 4's produced `uk.json` + `en.json` directly — those files define the canonical key names and structure (per Decision 8: nested namespaces, camelCase keys).
