# Task 9: Auth pages translation (5 files)

## Required Skills

Before starting, load:
- `/skill:code-writing` вАФ [SKILL.md](../../../.claude/skills/code-writing/SKILL.md)

## Description

Translate the five auth pages вАФ `pages/auth/login.vue`, `register.vue`, `forgot-password.vue`, `reset-password.vue`, `verify-email.vue` вАФ from hardcoded UA literals to `t()` calls backed by the locale dictionaries built in Task 4. Replace all hand-rolled HTTP-error-to-string mappings in `catch` blocks with the `useApiError()` factory composable created in Task 5. Switch every `<NuxtLink>` to `<NuxtLinkLocale>` and wrap every `navigateTo()` target with `localePath()` so links and redirects respect the active locale prefix under strategy `prefix_except_default`.

A subtle but important constraint (Decision 11): each Zod schema in these pages is constructed at `<script setup>` time and "bakes in" whatever string `t()` returned at that moment. Without restructuring, switching the locale live (via `<LangSwitcher>` from Task 6) leaves stale UA validation messages in the closure. To avoid that, the entire `toTypedSchema(z.object({...}))` chain must be wrapped in a single `computed(() => toTypedSchema(z.object({...})))` and passed directly to `useForm({ validationSchema: schemaComputed })` as a `ComputedRef`. vee-validate v4 reads the `ComputedRef` reactively and re-runs validation when `locale.value` changes. Do NOT call `unref()` / `.value` on the computed schema before passing it to `useForm` вАФ unwrapping breaks the reactive subscription and reproduces the stale-message bug. (This is the same reactivity contract that Task 10 follows for the dashboard/profile schemas.)

The `verify-email.vue` page additionally uses interpolated cooldown text (`–Ч–∞—З–µ–Ї–∞–є—В–µ ${cooldownSecondsLeft} —Б` and `–°–њ—А–Њ–±—Г–є—В–µ –Ј–љ–Њ–≤—Г —З–µ—А–µ–Ј {{ cooldownSecondsLeft }} —Б`). These become `t('auth.verifyEmail.cooldown', { seconds: cooldownSecondsLeft })` with proper named-placeholder interpolation in the locale JSON.

This task closes AC11 (auth pages localized), AC12 (Zod messages localized + locale-reactive), AC13 (API errors via shared composable), and contributes to the `<NuxtLink>` вЖТ `<NuxtLinkLocale>` and `navigateTo`-wrap acceptance criteria. Existing Vitest specs for these pages were authored against UA defaults and the UA literals stay 1:1 with what the tests assert (per Decision 16) вАФ but every spec covering a page modified by this task needs a small mock addition because the implementation forces a new auto-imported composable (`useLocalePath`) into `setup()` that the existing tests don't stub out.

**Authorized exceptions to AC18 / Decision 16.** Decision 16 originally authorized one exception (`auth.global.spec.ts`). This task adds **five more** authorized exceptions вАФ one for each of the five page specs in scope: `login.spec.ts`, `register.spec.ts`, `forgot-password.spec.ts`, `reset-password.spec.ts`, `verify-email.spec.ts`. The rule is **unconditional**: every spec gains a `mockNuxtImport('useLocalePath', () => () => (path: string) => path)` identity-passthrough mock at the top of the file (alongside any existing `mockNuxtImport` declarations). This mirrors the unconditional rule applied in Task 10 for dashboard/profile specs and removes the "only if it fails" ambiguity. Identity passthrough preserves uk-default behaviour where unprefixed paths are emitted as-is, so `expect()` assertions and `toHaveBeenCalledWith(...)` payloads stay green without modification. The exception is strictly limited to mocking the `useLocalePath` auto-import вАФ UA literals in `expect()` assertions, `toHaveBeenCalledWith(...)` payloads, screen-text expectations, and test names remain unchanged. Concretely: `register.spec.ts` currently asserts `navigateToMock.toHaveBeenCalledWith({ path: '/auth/login', query: { registered: '1' } })`; with identity-passthrough mock, the actual call `navigateTo({ path: localePath('/auth/login'), query: { registered: '1' } })` collapses back to the same payload, so the assertion stays green without modification.

- vee-validate v4 `useForm({ validationSchema })` accepts a `ComputedRef` / `Ref` directly and re-runs validation when the ref's value changes. **Pass `schemaComputed` to `useForm` as-is вАФ never `unref(schemaComputed)`, never `schemaComputed.value`** (unwrapping converts the reactive subscription into a one-time static value, reproducing the stale-message bug). **Implementation hints:**
- vee-validate v4 + `@vee-validate/zod` reactive schema pattern (per code-research ¬І10, mirrors Task 10's pattern):
  - Build the entire `toTypedSchema(z.object({...}))` chain inside a single `computed`:
    ```ts
    const schemaComputed = computed(() =>
      toTypedSchema(z.object({
        email: z.string().email(t('validation.emailFormat')),
        password: z.string().min(8, t('validation.passwordMin')),
      }))
    )
    ```
    For schemas with `.refine`, include the refine inside the same `computed`:
    ```ts
    const schemaComputed = computed(() =>
      toTypedSchema(
        z.object({ /* ... */ }).refine(
          (data) => data.password === data.confirmPassword,
          { message: t('validation.passwordMismatch'), path: ['confirmPassword'] }
        )
      )
    )
    ```
  - Pass the `ComputedRef` directly to `useForm`:
    ```ts
    const { handleSubmit } = useForm({ validationSchema: schemaComputed })
    ```
  - **Never** call `unref(schemaComputed)`, `schemaComputed.value`, or wrap in another `toTypedSchema(...)` вАФ vee-validate v4 reads the `ComputedRef` reactively, so unwrapping breaks the locale-switch behaviour. This is the exact pattern Task 10 settled on after its review round 2.
  - If you suspect `validationSchema` is not reading the ref reactively in your scenario, do NOT fall back to `unref()`; instead, query Context7 for vee-validate v4 + `@vee-validate/zod` reactive schema patterns and confirm before deviating.
- `useApiError()` factory call site:
  - Instantiate once at the top of `<script setup>`: `const apiError = useApiError()`.
  - Inside catch: `submitError.value = apiError(e, '<context>')` where `<context>` is `'login'` / `'register'` / `'resetPassword'`.
  - The factory internally walks `errors.<context>.<httpCode>` вЖТ `errors.<context>.generic` вЖТ `errors.generic` via `te()` chain.
- Cooldown button template in `verify-email.vue`: ternary inside `{{ }}` interpolation calling `t('auth.verifyEmail.cooldown', { seconds: cooldownSecondsLeft })` for `> 0` branch, `t('auth.verifyEmail.resend')` otherwise.
- Auto-imports under @nuxtjs/i18n v9: `useI18n`, `useLocalePath`, `<NuxtLinkLocale>` are auto-registered. Explicit `const { t } = useI18n()` is still recommended for IDE intellisense and for clarity in code review.
- When locale JSON keys are uncertain, cross-reference Task 4's produced `uk.json` + `en.json` directly вАФ those files define the canonical key names and structure (per Decision 8: nested namespaces, camelCase keys).
