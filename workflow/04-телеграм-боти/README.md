# 04 — Telegram-боти

## Мета
Дозволити користувачу підключити свого Telegram-бота до проекту: отримувати від нього вхідні повідомлення, надсилати вихідні, керувати станом.

## Користувацька цінність
Це реальний "ввід-вивід" сервісу. Без бота нічого не працює — ні підписники, ні воронки, ні розсилки.

---

## MUST HAVE

### Підключення бота
- Сторінка "Settings → Bot" у проекті
- Поле "Telegram Bot Token" + кнопка "Connect"
- При збереженні:
  1. Виклик `getMe` через Telegram API для валідації токена
  2. Якщо успіх → зберігаємо токен (зашифрованим at-rest), показуємо `@username`, ім'я бота, аватар
  3. Якщо помилка → зрозуміле пояснення ("Token invalid", "Token revoked")
- **Автоматичне налаштування webhook'а** на наш ендпоінт `/webhooks/telegram/{projectId}` із secret token у заголовку
- Перевірка унікальності: один і той же бот не може бути підключений у двох проектах одночасно

### Прийом повідомлень (incoming)
- Endpoint `POST /webhooks/telegram/{projectId}` приймає updates
- Перевірка secret token у заголовку `X-Telegram-Bot-Api-Secret-Token`
- Парсинг update: `message`, `callback_query`, `/start`, `/stop`
- Швидке збереження raw update у БД + постановка у Redis-чергу для асинхронної обробки
- Відповідь Telegram-серверу `200 OK` за < 100ms

### Обробка вхідних (worker)
- Окремий worker читає чергу і:
  - Знаходить або створює `Subscriber` (логіка в `05-підписники`)
  - Запускає тригери воронок (логіка в `06-воронки`)
  - Зберігає подію в `events` таблицю
- Idempotency: повторне отримання того самого `update_id` ігнорується

### Відправка повідомлень (outgoing)
Сервіс `TelegramSender` з методами:
- `sendText(chatId, text, options?)`
- `sendWithButtons(chatId, text, inlineButtons[])` — підтримка inline keyboards
- `sendImage(chatId, url|fileId, caption?)`
- `editMessage(chatId, messageId, text)` — для редагування
- `deleteMessage(chatId, messageId)`

**Обов'язкова поведінка:**
- Автоматичний retry з exponential backoff (до 3 спроб) при `5xx`, network errors
- Обробка rate-limit'ів від Telegram (`429 Too Many Requests` з `retry_after`) — чекаємо і повторюємо
- При помилці `403 Forbidden: bot was blocked by the user` — позначаємо підписника як `blocked`
- При `400 Bad Request: chat not found` — позначаємо як `deleted`

### Команди бота за замовчуванням
- `/start` — реєструє/оновлює підписника, запускає воронку з тригером "/start" (якщо є). Може приймати payload: `/start ref_123`
- `/stop` — позначає підписника як `unsubscribed`. Не отримуватиме розсилок. Тригерить вихід з активних воронок.

### Видалення / переконфігурація бота
- Кнопка "Disconnect bot":
  - Виклик `deleteWebhook` у Telegram
  - Очистити токен з БД
  - Підписники лишаються (для історії), нові не приходитимуть
- Можливість змінити токен (re-connect новим)

### Безпека
- Токен зберігається зашифрованим (AES-256, ключ у env)
- На UI токен показуємо masked: `1234567890:ABC***...***xyz`
- Можна "Show full token" один раз з підтвердженням пароля
- Webhook secret token генерується випадковий, унікальний на проект

### Тест-повідомлення
- Кнопка "Send test message" у налаштуваннях бота
- Вимагає, щоб користувач хоча б раз написав боту `/start` (треба знати його chat_id)
- Відправляє "Hello from Bot Funnel Service! Bot connected ✅"

## Що НЕ входить (свідомо)
- Декілька ботів в одному проекті — `12-nice-to-have`
- Telegram Login Widget для веб-сайтів — `12-nice-to-have`
- Bot Father command-set управління (set commands, set description, set menu) — `12-nice-to-have`
- Telegram Mini Apps / Web Apps — `12-nice-to-have`
- Підтримка груп / каналів — поки тільки 1-on-1 DM
- Голосові, відео, стікери як reply від бота — `12-nice-to-have` (поки тільки текст + кнопки + зображення)
- Telegram Payments — `12-nice-to-have`

---

## Acceptance criteria
1. Користувач вставляє валідний токен → бачить `@username` та ім'я свого бота
2. При невалідному токені — зрозуміла помилка
3. Користувач пише боту `/start` → з'являється підписник у проекті
4. Користувач відправляє тестове повідомлення з адмінки → отримує його в Telegram
5. Якщо підписник заблокував бота і ми спробували надіслати — статус `blocked` коректно записується
6. Той самий бот не можна підключити у двох проектах одночасно

## Технічні нотатки
- Webhook URL **обов'язково HTTPS** (Telegram вимагає)
- Для локальної розробки — підтримати ngrok або polling-режим у dev-конфізі
- Усі вхідні повідомлення зберігати в `events` для дебагу і майбутньої аналітики (з TTL 90 днів, щоб БД не розпухла)
- Шифрування токенів — використати, наприклад, libsodium / node:crypto AES-256-GCM
