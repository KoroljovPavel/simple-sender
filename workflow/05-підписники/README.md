# 05 — Підписники, теги, кастом-поля

## Мета
Зберігати інформацію про людей, які взаємодіють з ботом, і дозволити сегментувати їх для воронок та розсилок.

## Користувацька цінність
Без бази підписників немає кому слати, немає для кого будувати воронки. Це CRM-частина продукту.

---

## MUST HAVE

### Автоматична реєстрація підписника
- При першій взаємодії з ботом (будь-яке повідомлення або `/start`):
  - Створюється `Subscriber` у проекті
  - Зберігаємо: `telegram_chat_id`, `telegram_user_id`, `username`, `first_name`, `last_name`, `language_code`, `is_premium` (Telegram-флаг)
- При повторних взаємодіях — оновлюємо `last_seen_at` і застарілі поля (якщо username змінився)
- Поле `subscribed_at` — фіксується один раз на створенні

### Статуси підписника
- `active` — нормальний стан, приймає повідомлення
- `unsubscribed` — написав `/stop` або відписався через кнопку. Не отримує розсилок, не запускається у воронках.
- `blocked` — заблокував бота (отримали 403 від Telegram). Не отримує нічого.
- `deleted` — Telegram-чат видалений (отримали 400 chat not found). Зберігається для історії.

### Список підписників (UI + API)
- Сторінка `Subscribers` у проекті
- Таблиця з колонками: ім'я / username / теги (badges) / дата підписки / статус
- **Пошук** по `first_name`, `last_name`, `username` (з мін. 2 символів)
- **Фільтри:** по тегу, по статусу, по даті підписки (range)
- Пагінація (50 на сторінку, cursor-based для API)
- Сортування за датою підписки і last_seen

### Профіль підписника
- Картка з усіма полями
- Список тегів (можна додавати / видаляти)
- Список custom fields зі значеннями (можна редагувати)
- Історія взаємодій (останні 50 подій: повідомлення, запуски воронок, отримані розсилки)
- Кнопки:
  - Add tag / Remove tag
  - Edit custom field
  - Manually unsubscribe
  - Send personal message (опц., в окремий діалог)

### Теги
- Тег — це рядок-slug (`vip`, `lead`, `paid_2024`, `course_buyer`)
- Список доступних тегів — на рівні проекту
- Створення тега:
  - Автоматично при першому додаванні (за принципом `find or create`)
  - Або явно через сторінку "Tags management" у проекті
- На підписнику — many-to-many зв'язок
- API: `add_tag`, `remove_tag`, `get_subscribers_by_tag`
- Сторінка управління тегами: список, кількість підписників на тегу, перейменувати, видалити (з підтвердженням)

### Custom Fields (схема + значення)
- На рівні проекту визначається схема:
  - `name` (slug, наприклад `city` або `purchase_amount`)
  - `label` (для UI: "Місто" або "Сума покупки")
  - `type`: `string` / `number` / `boolean` / `date`
  - `default_value` (опц.)
- На підписнику зберігається JSONB з ключами і значеннями
- Можна редагувати:
  - У UI вручну
  - Через крок воронки `Set Custom Field`
  - Через API
- Ліміт: до 20 полів на проект (м'який, у конфізі)

### Сегменти (legkий filter)
- Сегмент = "усі підписники з тегом X" або "усі без тегу X" або "усі активні"
- Формат для API: `{ "status": "active", "tags_include": ["vip"], "tags_exclude": ["unsubscribed_promo"] }`
- На MVP — без AND/OR-конструкторів, без фільтрів за custom fields

### Unsubscribe
- Команда `/stop` від підписника → статус `unsubscribed`, м'який вихід з активних воронок
- У broadcast-повідомленнях додати кнопку "Unsubscribe" (опц., можна винести в `12-nice-to-have`)
- Підписник може повернутись через нове `/start` → статус знову `active` (з тими самими тегами)

### Експорт
- Кнопка "Export CSV" на сторінці підписників
- CSV з полями: id, telegram_id, name, username, tags (через ;), custom fields (як JSON-рядок), status, subscribed_at, last_seen_at
- Стандартний UTF-8 з BOM (для Excel)
- Респектує поточні фільтри

---

## Що НЕ входить (свідомо)
- Складний segment-builder з AND/OR/NOT і умовами по custom fields — `12-nice-to-have`
- Імпорт CSV — `12-nice-to-have`
- Lead scoring — `12-nice-to-have`
- Об'єднання дублікатів між каналами — `12-nice-to-have` (бо MVP моноканал)
- GDPR-запити (експорт усіх даних користувача за email, право на видалення) — `12-nice-to-have`
- Автоматичні bulk-дії (масове додавання тегу за фільтром) — `12-nice-to-have`

---

## Acceptance criteria
1. Користувач пише боту → з'являється у списку підписників із заповненими ім'ям, username
2. Можна додати/видалити тег у профілі — зберігається
3. Фільтр по тегу повертає правильний набір
4. CSV-експорт містить усіх підписників проекту з тегами і полями
5. `/stop` → статус `unsubscribed`, користувач не отримує наступну розсилку
6. Кастом-поле з типом `number` приймає тільки числа (валідація)

## Технічні нотатки
- Індекси: `(project_id, telegram_user_id)` UNIQUE — щоб уникнути дублів
- JSONB для custom fields — швидко, гнучко, можна індексувати окремі ключі при потребі
- Таблиця `subscriber_events` для історії — append-only, з retention 1 рік
