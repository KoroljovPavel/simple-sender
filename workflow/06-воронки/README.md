# 06 — Воронки (Flows)

## Мета
Дозволити будувати автоматичні сценарії спілкування з підписниками: "коли стається X — відправ цю послідовність повідомлень з гілками і діями".

## Користувацька цінність
**Серце продукту.** Це те, заради чого користувач прийшов. Усе інше — допоміжне.

---

## Концепція (MVP)
Воронка — упорядкований **список кроків**. Кроки виконуються по черзі. Деякі кроки розгалужують потік за натисканням кнопки. Без drag-and-drop редактора — простий step-by-step UI: "додати крок", "перемістити вгору/вниз", "видалити".

Це state machine: для кожного підписника зберігається `current_step_index` і `status`.

---

## MUST HAVE

### CRUD воронок
- **Створити воронку:** назва, опис (опц.), статус (`draft` / `active` / `paused`)
- **Редагувати** склад кроків
- **Дублювати** воронку (для створення варіацій)
- **Видалити** воронку (з підтвердженням, скасовує всі активні execution'и)
- **Список воронок** у проекті з фільтром за статусом

### Типи кроків (мінімум)
1. **Send Message** — текст (до 4096 символів) з опц. inline-кнопками
2. **Send Image** — URL зображення або завантаження + caption
3. **Wait (Delay)** — затримка: 5 секунд / 1 хв / 5 хв / 1 година / 24 години / кастомний інтервал у хвилинах
4. **Wait for Reply** — пауза до відповіді підписника
   - Може приймати: натиснуту кнопку (callback) або текст
   - Має таймаут (за замовчуванням 24 години)
   - Кнопки породжують гілки (Branch)
5. **Add Tag** — додати тег підписнику (з вибору або новий)
6. **Remove Tag** — прибрати тег
7. **Set Custom Field** — присвоїти значення (літерал або плейсхолдер `{user.first_name}`)
8. **Branch by Button** — розгалуження за натиснутою кнопкою у попередньому Wait for Reply
9. **End Funnel** — явне завершення (опц., бо просто закінчення списку = завершення)

### Тригери запуску воронки
Один до одного: `trigger_type` + `trigger_value`.

- **`/start` command** — підписник написав `/start` (опц. з payload, наприклад `/start ref_promo`)
- **Keyword** — повідомлення містить слово (case-insensitive, регулярка опційно в nice-to-have)
- **Tag added** — підписнику додано конкретний тег (вручну, через API, через іншу воронку)
- **API event** — POST на `/api/v1/events` з певним `event_name`

**Anti-spam:** один підписник не може запустити ту саму воронку повторно, поки попередня не завершилась. Можна обійти прапорцем `allow_re_enter: true` на воронці.

### Шаблонізація змінних
- У текстах повідомлень підтримати:
  - `{user.first_name}` / `{user.last_name}` / `{user.username}`
  - `{custom.field_name}` — кастом-поля підписника
- Невідома або порожня змінна → пустий рядок (без падіння виконання)
- Екранування: `{{` як літерал `{`

### Engine (виконання)
- Окрема таблиця `funnel_executions`:
  - `subscriber_id`, `funnel_id`, `current_step_id`, `status`, `started_at`, `next_run_at`, `last_button_clicked`
- Статуси execution: `running` / `waiting` / `completed` / `cancelled` / `failed`
- При `Wait` крок планує `next_run_at` у БД
- **Шедулер** (BullMQ або pg-boss) раз на 30 секунд забирає прострочені execution'и і виконує наступний крок
- При надходженні відповіді (callback_query / message) → шукаємо активний execution з `status: waiting_for_reply` і просуваємо

### Idempotency і відмовостійкість
- Якщо worker впав посеред кроку — після рестарту не повторити повідомлення двічі
- Кожен крок має проміжний статус: `pending → in_progress → done`
- Worker'и беруть execution'и з блокуванням (SELECT FOR UPDATE SKIP LOCKED)

### Зупинка / відписка
- Якщо підписник пише `/stop` → усі його активні execution маркуються `cancelled`
- Якщо адмін зупинив воронку (`status: paused`) → нові execution не створюються, активні дограють до кінця
- Кнопка "Force stop all running executions" у налаштуваннях воронки (з підтвердженням)

### Тестування воронки
- Кнопка "Test for me" — запускає воронку на твоєму особистому Telegram
- Потрібно один раз "пов'язати" акаунт сервісу з Telegram (написати боту з кодом — як у Manychat і подібних)

### UI редактора воронки
- Вертикальний список кроків
- Кнопка "+ Add step" відкриває picker типу кроку
- На кожному кроці — кнопки "вгору", "вниз", "видалити", "редагувати"
- Inline-редагування або модалка для складніших типів (Wait for Reply з кнопками)
- Превью повідомлень (як виглядатиме в Telegram)
- Валідація перед збереженням: усі обов'язкові поля заповнені, гілки коректні

### Аналітика per-execution
- На сторінці воронки у вкладці "Statistics" (детально в `09-аналітика`):
  - Скільки запущено / в процесі / завершено / скасовано
  - Drop-off на кожному кроці
  - Розподіл по гілках

## Що НЕ входить (свідомо)
- **Drag-and-drop візуальний редактор з canvas** — `12-nice-to-have` (це окрема велика задача на 4-6 тижнів)
- A/B тести (розщеплення трафіку 50/50) — `12-nice-to-have`
- Conditional branching за custom fields і тегами (зараз тільки за натиснутою кнопкою) — `12-nice-to-have`
- Math-expressions і скрипти всередині кроків — `12-nice-to-have`
- HTTP-крок (виклик зовнішнього API з воронки) — `12-nice-to-have`
- AI-крок (відповідь через LLM) — `12-nice-to-have`
- Маркетплейс шаблонів воронок — `12-nice-to-have`
- Розклад "запускати воронку щодня о 10:00" — `12-nice-to-have`

---

## Acceptance criteria
1. Користувач створює воронку з 5 кроків (Send → Wait 1 хв → Send з 2 кнопками → 2 гілки)
2. `/start` у бота → підписник проходить воронку, відповідає, потрапляє в правильну гілку
3. У повідомленні бачить підставлене своє ім'я
4. Якщо натиснув `/stop` посеред — воронка зупиняється
5. Якщо worker рестартував — повідомлення не дублюється
6. Воронка зі статусом `paused` не створює нових execution'ів

## Технічні нотатки
- Engine — це state machine. Поточний крок зберігається в БД, переходи події-керовані
- Усі `Wait` виконуються через scheduled jobs (BullMQ delayed jobs або pg-boss)
- Для production масштабу — окремий worker-процес, не змішувати з API
- При великій базі (>100k підписників) — індекси на `funnel_executions(status, next_run_at)` критичні
- **Цей епік великий — рекомендую розбити на підзадачі:** (1) CRUD + базові кроки, (2) Engine + scheduler, (3) Wait for Reply + Branch, (4) Тригери, (5) UI редактора
