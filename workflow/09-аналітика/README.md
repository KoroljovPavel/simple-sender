# 09 — Базова аналітика

## Мета
Показати користувачу в цифрах: скільки в нього підписників, як працюють воронки, як пройшли розсилки.

## Користувацька цінність
Без аналітики неможливо зрозуміти, чи воронка дійсно працює, де люди відвалюються, чи розсилки доходять. Це must для прийняття бізнес-рішень користувача.

---

## Принципи MVP-аналітики
- **3-5 важливих чисел** замість 50 другорядних
- Дашборд проекту → одразу зрозуміло на 5 секунд
- Графіки прості: stacked bar / line, без custom date pickers
- Без real-time — оновлення раз на хвилину (Redis cache)

---

## MUST HAVE

### Дашборд проекту (головна сторінка)
**4 ключові метрики зверху (KPI cards):**
1. Total subscribers (active)
2. New subscribers за останні 7 днів (з трендом ↑/↓)
3. Active funnels (скільки воронок зараз `active`)
4. Messages sent за 7 днів (broadcast + funnel)

**2 графіки під ними:**
- New subscribers by day за останні 30 днів (bar chart)
- Messages sent by day за останні 30 днів (line chart, дві лінії: broadcast і funnel)

**Список останніх дій:**
- 10 останніх подій: створено воронку, запущено розсилку, тощо

### Аналітика воронки
На сторінці кожної воронки вкладка "Statistics":

**Загальні цифри:**
- Total executions (всього запусків)
- Currently running
- Completed
- Cancelled (через `/stop` або вручну)
- Failed (помилки виконання)
- **Completion rate** = completed / (completed + cancelled + failed) × 100%

**Drop-off на кроках:**
- Список кроків з кількістю підписників, які дійшли до цього кроку
- Візуалізація: горизонтальна шкала "100% → 80% → 60%"
- Підсвічений найгірший крок (де максимальний drop)

**Розподіл по гілках:**
- Для кожного `Wait for Reply` зі стилем кнопок — скільки підписників натиснули кожну
- Bar chart: button A — 60%, button B — 40%

### Аналітика розсилки
На сторінці кожної розсилки:

**Цифри:**
- Total recipients
- Sent (доставлено успішно)
- Failed (з retry-помилками)
- Blocked (юзер заблокував бота)
- Unsubscribed (були вже відписані — пропущено)
- **Delivery rate** = sent / total × 100%

**Час:**
- Started at, completed at
- Average throughput (msg/sec)
- Total duration

**Список помилок:**
- Перші 100 невдалих з причиною (для дебагу)
- Можливість експорту в CSV

### Період і фільтри
- Перемикач 7 днів / 30 днів / 90 днів — глобально для дашборду
- Без custom date range на MVP

### Глобальна аналітика для Super Admin
Деталі в `10-адмін-панель`. Окремий дашборд по всій платформі.

## Що НЕ входить (свідомо)
- Cohort analysis ("підписники за червень — як вони конвертують?") — `12-nice-to-have`
- Conversion funnel за UTM джерелами — `12-nice-to-have`
- Click-tracking на посиланнях у повідомленнях — `12-nice-to-have`
- Real-time дашборд (websocket-оновлення) — `12-nice-to-have`
- Експорт звітів у PDF / Excel — `12-nice-to-have`
- Інтеграції з GA4 / Yandex Metrika — `12-nice-to-have`
- Custom dashboards (юзер сам збирає) — `12-nice-to-have`
- Прогнози / ML / "best time to send" — `12-nice-to-have`

---

## Acceptance criteria
1. Користувач відкриває проект і одразу бачить 4 ключові цифри (loading < 1с)
2. На сторінці воронки видно скільки людей завершили і де відвалюються
3. На сторінці розсилки видно реальні sent/failed і причини помилок
4. Зміна періоду 7д ↔ 30д ↔ 90д оновлює всі цифри і графіки
5. Аналітика підвантажується з кешу (Redis), оновлюється раз на хв

- `07-розсилки` (метрики розсилок)

---

## Технічні нотатки
- Усі метрики обчислюються з основних таблиць + допоміжних агрегатів (materialized views або agg-таблиць оновлюваних cron'ом)
- Для дашборду — кеш 60 секунд (Redis), щоб не довбати БД при кожному відкритті
- Можна винести в окремий read-replica Postgres у майбутньому, поки одна БД
- Графіки на фронті — recharts або chart.js, без важких BI-бібліотек
