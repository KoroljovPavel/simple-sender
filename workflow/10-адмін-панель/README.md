# 10 — Панель Super Admin

## Мета
Дати власнику сервісу інструмент моніторингу і модерації всіх користувачів і проектів. Без цього неможливо реагувати на зловживання, дебажити проблеми клієнтів, бачити стан бізнесу.

## Користувацька цінність
Користувач Super Admin — це 1-2 особи (засновник + cto/девопс). Інструмент для них.

---

## MUST HAVE

### Доступ
- Окремий маршрут: `/admin/*` (на тому ж домені)
- Доступ тільки для користувачів з прапорцем `is_super_admin = true` у БД
- Прапорець ставиться seed-скриптом або вручну через консоль БД (без UI для надання адмінських прав — anti-abuse)
- Окремий middleware/guard на всі `/admin` маршрути перевіряє прапорець
- Звичайний user, що зайде за `/admin` URL → 403, без натяку що сторінка взагалі існує

### Список користувачів (`/admin/users`)
- Таблиця всіх зареєстрованих користувачів сервісу
- **Колонки:** email, ім'я, статус (`active` / `pending` / `blocked`), дата реєстрації, к-сть проектів, остання активність
- **Пошук** по email (з мін. 3 символів)
- **Фільтри:** статус, дата реєстрації (range), чи має проекти
- Сортування за датою реєстрації, last_login
- Пагінація (50 на сторінку)

**Дії над користувачем:**
- Переглянути деталі (повна картка користувача без логіну в його акаунт)
- Заблокувати → статус `blocked`, не може логінитись (і відразу інвалідуються активні токени)
- Розблокувати → статус повертається в `active`
- Soft-delete акаунт (з усіма проектами)
- Resend verification email (якщо `pending`)

### Список проектів (`/admin/projects`)
- Таблиця всіх проектів усіх користувачів
- **Колонки:** назва, власник (email), к-сть підписників, к-сть активних воронок, дата створення, статус
- Фільтри: за власником, за статусом, по даті
- Пошук по назві проекту або email власника
- Перехід на read-only сторінку проекту по кліку

**Дії над проектом:**
- Видалити проект (з підтвердженням) — каскадно прибирає весь контент
- Переглянути read-only (бачимо налаштування, воронки, підписників, але не редагуємо нічого)

### Глобальний дашборд (`/admin`)
**KPI cards:**
- Total registered users
- Active users (хто логінився за 30 днів)
- Total projects (active / soft-deleted)
- Total subscribers across all projects
- Messages sent за 24 години (broadcast + funnel)
- Errors за 24 години

**Графіки:**
- User growth за 30 днів (нові реєстрації по днях)
- Projects growth за 30 днів
- Messages sent за 7 днів (по днях)

### Журнал подій / Audit log (`/admin/audit`)
- Останні N подій по всій системі
- **Типи подій:**
  - `user.registered` / `user.login` / `user.login_failed`
  - `user.blocked` / `user.unblocked` (хто заблокував)
  - `project.created` / `project.deleted`
  - `bot.connected` / `bot.disconnected`
  - `admin.action` — будь-яка дія Super Admin
  - `system.error` (критичні помилки)
- Фільтри: тип події, користувач, дата
- Зберігати мінімум 90 днів

### Системний стан (`/admin/system`)
- Статус черг BullMQ:
  - Розмір кожної черги (waiting, active, delayed, failed)
  - К-сть failed jobs за 24 год
  - Кнопка "Retry all failed jobs"
- Статус Telegram-ботів:
  - Скільки активних
  - Скільки з помилками webhook'у за останню годину
  - Список ботів з проблемами (для дебагу)
- Health check всіх сервісів: Postgres, Redis, Telegram API доступність
- К-сть API-викликів за останню годину (з користувацьких ключів)

### Безпека
- Усі дії Super Admin'а логуються в окрему таблицю `admin_actions`:
  - `actor_id`, `action_type`, `target_type`, `target_id`, `details` (JSONB), `timestamp`, `ip_address`
- На MVP — достатньо сильного паролю + IP-allowlist (опц.)
- 2FA для Super Admin — `12-nice-to-have` (бажано, але не блокер для MVP)

---

## Що НЕ входить (свідомо)
- Імперсонація (логін під чужим акаунтом для саппорту) — `12-nice-to-have`
- Білінг і фінансова статистика — `12-nice-to-have` (поки немає платних тарифів)
- Розсилки від адмінів усім користувачам сервісу — `12-nice-to-have`
- Система саппорту / тікети — `12-nice-to-have`
- Feature flags — `12-nice-to-have`
- Custom roles для адмінів (super / billing / support) — `12-nice-to-have`
- Експорт даних користувача за GDPR-запитом через UI — `12-nice-to-have`

---

## Acceptance criteria
1. Super Admin логіниться, бачить `/admin`, звичайний user туди не зайде (403)
2. Видно список усіх користувачів і всіх проектів
3. Бан користувача → user не може логінитись, бачить пояснення
4. Видалення проекту → каскадно прибирається весь контент
5. Усі дії адміна записуються в audit log з IP та timestamp
6. Дашборд показує реальні цифри по платформі

---

## Залежності
- `02-автентифікація` (для is_super_admin прапорця)
- Усі інші епіки (для перегляду їхнього вмісту)

---

## Технічні нотатки
- Окремий React-роутер для `/admin`, окремий layout, інше меню
- Можна винести в окремий subdomain `admin.yoursite.com` пізніше (`12-nice-to-have`)
- БД-запити для глобальних метрик можуть бути важкими — кешувати в Redis на 5-10 хв
- На security-чутливі дії (delete project, block user) — логування + email-нотифікація іншим Super Admin'ам
